# Task 7.4 -- WebSocket Session Streaming and Notifications

## Objective

Implement WebSocket endpoints for real-time session activity streaming and system notifications, enabling the web dashboard to show live session output and instant gate/alert notifications.

## Context

The web dashboard needs real-time updates for two use cases: (1) watching a live session's activity in a session panel, and (2) receiving instant notifications about new gates, alerts, and status changes. Bun.serve natively supports WebSocket upgrades via the `websocket` option — no additional packages needed.

Depends on 7.1 (router and auth). Used by 7.7-7.9 (frontend pages).

## Requirements

- [ ] `WS /api/ws/sessions/:id/stream` -- real-time stream of session activity
- [ ] Session stream events: `{ type: "status_change" | "cost_update" | "text", data: any, timestamp: string }`
- [ ] `WS /api/ws/notifications` -- system-wide notifications
- [ ] Notification events: `{ type: "gate_pending" | "gate_resolved" | "session_started" | "session_completed" | "session_failed" | "budget_warning" | "alert", data: any, timestamp: string }`
- [ ] `NanoClawEvents` class: lightweight pub/sub for internal event dispatching
- [ ] `NanoClawEvents.emit(event, data)` publishes to all subscribed WebSocket connections
- [ ] `NanoClawEvents.subscribe(channel, ws)` and `unsubscribe(channel, ws)` for connection management
- [ ] Auth validated on WebSocket upgrade request (same CF Access JWT check from 7.1)
- [ ] Connection tracking: `Set<ServerWebSocket>` per channel, cleaned up on close
- [ ] Heartbeat: server sends `{ type: "ping" }` every 30 seconds; client expected to respond with pong
- [ ] WebSocket upgrade integrated into the router's fetch handler (check `Upgrade` header before route matching)
- [ ] Export `WsMessageType` and `WsMessage` types
- [ ] Barrel export from `src/api/index.ts`

## Interface Contract

**Inputs:** WebSocket upgrade requests. Internal events from NanoClaw subsystems.

**Outputs:** `NanoClawEvents` class for internal event publishing. WebSocket message types.

**Side Effects:** Maintains open WebSocket connections. Sends real-time messages to connected clients.

## Technical Approach

```typescript
// src/api/ws.ts
export class NanoClawEvents {
  private channels = new Map<string, Set<ServerWebSocket<WsData>>>();
  
  emit(channel: string, event: WsMessage): void {
    const subscribers = this.channels.get(channel);
    if (!subscribers) return;
    const msg = JSON.stringify(event);
    for (const ws of subscribers) {
      ws.send(msg);
    }
  }
  
  subscribe(channel: string, ws: ServerWebSocket<WsData>): void { ... }
  unsubscribe(channel: string, ws: ServerWebSocket<WsData>): void { ... }
}
```

The router's fetch handler checks for the `Upgrade: websocket` header first. If present, it matches the WS patterns, validates auth, and calls `server.upgrade(req, { data: { channel, sessionId } })`. The `websocket` handlers on `Bun.serve` manage open/close/message events.

For session streaming: downstream integration (task 7.10) will call `events.emit(`session:${sessionId}`, ...)` from the session runner's status change points. This task builds the infrastructure; the actual event emission points are wired in 7.10.

## What NOT to Do

- Do not implement the actual event emission from session runners -- that is task 7.10
- Do not add reconnection logic server-side -- that is the client's responsibility (task 7.6)
- Do not use a third-party WebSocket library -- Bun.serve has native support
- Do not buffer messages for disconnected clients -- real-time only

## Recovery Guidance

- If Bun's WebSocket API is tricky to test, test the NanoClawEvents pub/sub independently (it's just a Map of Sets)
- The WebSocket upgrade in Bun.serve requires returning `undefined` from fetch and using `server.upgrade(req, { data })` -- see Bun docs
- Focus on the pub/sub infrastructure first; the actual event emission points come in 7.10
