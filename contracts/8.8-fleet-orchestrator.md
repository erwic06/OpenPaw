# Task 8.8 -- Fleet Orchestrator and NanoClaw Integration

## Objective

Wire all fleet subsystems (loader, scheduler, event triggers, routing, adapters) into a `FleetOrchestrator` class and integrate it into the NanoClaw startup sequence in `src/index.ts`.

## Context

The final integration task for Phase 8 (analogous to 3.9, 5.6, 7.10). Individual fleet modules exist from tasks 8.1-8.7 but are not yet connected to the running daemon. This task creates the `FleetOrchestrator` that owns the lifecycle of all fleet subsystems and wires it into `src/index.ts`.

Depends on 8.3 (loader), 8.4 (ServiceAdapter), 8.5 (scheduler), 8.6 (events), 8.7 (routing).

## Requirements

- [ ] `FleetOrchestrator` class with `FleetOrchestratorDeps` DI interface
- [ ] `FleetOrchestratorDeps`: `{ db, secrets, sendMessage, chatId, repoDir, agentsDir, anthropicApiKey, openaiApiKey, geminiApiKey, alertSystem?, budgetEnforcer?, events?: NanoClawEvents }`
- [ ] `start(): void` -- loads definitions, starts scheduler, starts event system, starts file watcher
- [ ] `stop(): void` -- stops scheduler, event system, file watcher
- [ ] `triggerAgent(def: AgentDefinition, triggeredBy: string, detail: string): Promise<void>`:
  1. Budget check via optional BudgetEnforcer
  2. Insert `agent_runs` row with status "running"
  3. Select correct adapter based on `adapterConfig.type`:
     - `llm`: route by provider/model to LLMAdapter/CodexAdapter/GeminiAdapter
     - `service`: create ServiceAdapter with service config
  4. Call `adapter.trigger(input)`
  5. Poll `adapter.status()` until terminal state
  6. Call `adapter.output()` to get results
  7. Call `routeOutput()` from task 8.7
  8. Update `agent_runs` with final status and `output_routed_to`
- [ ] File watcher onChange: re-sync definitions to DB, update scheduler and event system
- [ ] Optional `events` dep: emit agent_started/agent_completed notifications for webapp
- [ ] Graceful error handling: adapter failures update agent_runs to "failed", don't crash orchestrator
- [ ] Concurrent agent runs: triggerAgent runs in background (fire-and-forget), orchestrator tracks active runs

## Interface Contract

**Inputs:** FleetOrchestratorDeps (DB, secrets, messaging, adapters).

**Outputs:** `FleetOrchestrator` class.

**Side Effects:** Agent sessions executed, output routed, DB updated, notifications emitted.

## Technical Approach

```typescript
// src/fleet/orchestrator.ts
export class FleetOrchestrator {
  private loader: ReturnType<typeof watchAgentDefinitions> | null = null;
  private scheduler: CronScheduler;
  private eventSystem: EventTriggerSystem;
  
  constructor(private deps: FleetOrchestratorDeps) {
    const triggerFn = this.triggerAgent.bind(this);
    this.scheduler = new CronScheduler({ db: deps.db, triggerAgent: triggerFn });
    this.eventSystem = new EventTriggerSystem({ db: deps.db, triggerAgent: triggerFn, repoDir: deps.repoDir });
  }
  
  start(): void {
    const defs = loadAllAgentDefinitions(this.deps.agentsDir);
    syncDefinitionsToDb(this.deps.db, defs);
    this.scheduler.start();
    this.eventSystem.start();
    this.loader = watchAgentDefinitions(this.deps.agentsDir, (updated) => {
      syncDefinitionsToDb(this.deps.db, updated);
      this.scheduler.updateSchedule(updated);
      this.eventSystem.updateDefinitions(updated);
    });
  }
}
```

In `src/index.ts`:
```typescript
const orchestrator = new FleetOrchestrator({
  db, secrets, sendMessage, chatId: telegramChatId,
  repoDir: REPO_DIR, agentsDir: `${REPO_DIR}/agents`,
  anthropicApiKey, openaiApiKey, geminiApiKey,
  alertSystem, budgetEnforcer, events,
});
orchestrator.start();

// In SIGTERM handler:
orchestrator.stop();
```
