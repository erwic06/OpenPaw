# Task 6.4 -- Telegram Command Dispatch

## Objective

Create a command dispatch layer that registers a Telegram message handler, parses incoming `/` commands using the parser from 6.1, and routes them to the appropriate intake handler callbacks.

## Context

The existing Telegram module (`src/messaging/telegram.ts`) provides `onMessage(handler)` to register message handlers. The HITL gate handler (`src/gates/index.ts:195`) already skips messages starting with `/` (`if (trimmed.startsWith("/")) return;`), so a separate command handler registered via `onMessage` will handle commands without conflicting with gate resolution.

Depends on 6.1 (command parser). Blocks 6.5, 6.6, 6.7, 6.8.

## Requirements

- [ ] `DispatchDeps` interface with DI for all 6 command handlers and messaging:
  - `onMessage: (handler: (chatId: number, text: string) => void | Promise<void>) => void`
  - `sendMessage: (chatId: number | string, text: string, options?: { parseMode?: string }) => Promise<void>`
  - `chatId: number`
  - `onResearch: (cmd: { depth: number; prompt: string }) => Promise<void>`
  - `onDeepResearch: (cmd: { prompt: string }) => Promise<void>`
  - `onProject: (cmd: { prompt: string }) => Promise<void>`
  - `onCoding: (cmd: { projectName: string; prompt: string }) => Promise<void>`
  - `onStatus: () => Promise<void>`
  - `onAgents: () => Promise<void>`
- [ ] `initCommandDispatch(deps: DispatchDeps): void` function
- [ ] Registered handler: parse text with `parseCommand()`, dispatch to matching handler
- [ ] On parse failure for a `/` message: send error message listing available commands
- [ ] On handler error: send error message to user, log to console
- [ ] Non-`/` messages ignored (fall through to gate handler)
- [ ] Exported from `src/intake/index.ts` barrel

## Interface Contract

**Inputs:** Raw Telegram messages via `onMessage` handler registration.

**Outputs:** Calls to the appropriate handler callback per parsed command.

**Side Effects:** Registers a message handler via `onMessage`. Sends error messages on parse failure.

## Technical Approach

```typescript
// src/intake/dispatch.ts
import { parseCommand } from "./parser.ts";
import type { ParsedCommand } from "./types.ts";

export interface DispatchDeps {
  onMessage: (handler: (chatId: number, text: string) => void | Promise<void>) => void;
  sendMessage: (chatId: number | string, text: string, options?: { parseMode?: string }) => Promise<void>;
  chatId: number;
  onResearch: (cmd: { depth: number; prompt: string }) => Promise<void>;
  onDeepResearch: (cmd: { prompt: string }) => Promise<void>;
  onProject: (cmd: { prompt: string }) => Promise<void>;
  onCoding: (cmd: { projectName: string; prompt: string }) => Promise<void>;
  onStatus: () => Promise<void>;
  onAgents: () => Promise<void>;
}

export function initCommandDispatch(deps: DispatchDeps): void {
  deps.onMessage(async (chatId, text) => {
    const trimmed = text.trim();
    if (!trimmed.startsWith("/")) return; // not a command, let gate handler deal with it

    const parsed = parseCommand(trimmed);
    if (!parsed) {
      await deps.sendMessage(deps.chatId,
        "Unknown command or invalid syntax.\n\nAvailable commands:\n" +
        "/research <depth> <prompt>\n/dr <prompt>\n/project <prompt>\n" +
        "/coding <project-name> <prompt>\n/status\n/agents"
      );
      return;
    }

    try {
      switch (parsed.command) {
        case "research": return await deps.onResearch(parsed);
        case "dr": return await deps.onDeepResearch(parsed);
        case "project": return await deps.onProject(parsed);
        case "coding": return await deps.onCoding(parsed);
        case "status": return await deps.onStatus();
        case "agents": return await deps.onAgents();
      }
    } catch (err) {
      const msg = err instanceof Error ? err.message : String(err);
      console.error(`[dispatch] command ${parsed.command} failed: ${msg}`);
      await deps.sendMessage(deps.chatId, `Command failed: ${msg}`);
    }
  });
}
```

The gate handler's existing `/` skip (line 195 in `src/gates/index.ts`) means both handlers see every message, but commands are only processed by the dispatch handler. No modification needed to `telegram.ts` or `gates/index.ts`.
