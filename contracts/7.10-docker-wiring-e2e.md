# Task 7.10 -- Docker Wiring, Vercel Deploy, and E2E Verification

## Objective

Wire all API routes and WebSocket handlers into `src/index.ts`, replacing the bare health endpoint. Configure Vercel deployment for the frontend. Verify end-to-end functionality.

## Context

The final integration task for Phase 7. Individual API modules exist from tasks 7.1-7.4 but are not yet connected to the running NanoClaw daemon. This task replaces the `Bun.serve` health endpoint with the full API router, wires WebSocket handling, and connects the NanoClawEvents to existing subsystems (AlertSystem, gate flows, session runners).

Depends on 7.3 (all routes), 7.4 (WebSocket), 7.5 (CF infrastructure), 7.9 (all frontend pages).

## Requirements

- [ ] Replace `Bun.serve` health endpoint in `src/index.ts` with full API router from task 7.1
- [ ] Construct `ApiDeps` from existing initialized subsystems (db, gates, runners, budget, alerts)
- [ ] Register all routes via `allRoutes(deps)` from task 7.3
- [ ] Add WebSocket upgrade handling to `Bun.serve` using `websocket` option
- [ ] Construct `NanoClawEvents` and wire event emission into:
  - AlertSystem: emit notification on alert send
  - Gate system: emit notification on gate create and resolve
  - SessionRunner: emit session status changes to session stream channels
  - ResearchRunner: emit research session status changes
- [ ] Auth middleware applied to all `/api/*` routes (read `cf_team_domain` and `cf_audience_tag` from secrets)
- [ ] `GET /health` remains unauthenticated (health checks don't go through CF Access)
- [ ] CORS headers for development: `Access-Control-Allow-Origin: *` when auth bypass is active
- [ ] Vercel configuration at `web/vercel.json` with build settings
- [ ] Docker build succeeds with all changes: `docker compose build`
- [ ] SIGTERM handler includes cleanup of WebSocket connections

## Interface Contract

**Inputs:** All Phase 7 modules, existing NanoClaw subsystems.

**Outputs:** Fully wired NanoClaw API server, Vercel deployment config.

**Side Effects:** NanoClaw now serves REST API and WebSocket connections on port 9999.

## Technical Approach

```typescript
// src/index.ts changes
import { createRouter, validateCfAccess, NanoClawEvents } from "./api/index.ts";
import { allRoutes } from "./api/routes/index.ts";

// After existing initialization...
const events = new NanoClawEvents();

const apiDeps: ApiDeps = {
  db,
  resolveGateFn: resolveGateById,
  events,
  // ... other deps from existing initialization
};

const router = createRouter(allRoutes(apiDeps), apiDeps);

const server = Bun.serve({
  port: HEALTH_PORT,
  async fetch(req, server) {
    // Health check (unauthenticated)
    if (new URL(req.url).pathname === "/health") {
      return new Response("ok");
    }
    
    // WebSocket upgrade
    if (req.headers.get("upgrade") === "websocket") {
      // ... validate auth, extract channel, call server.upgrade()
    }
    
    // Auth check for API routes
    const cfTeamDomain = secrets.get("cf_team_domain");
    if (cfTeamDomain) {
      const valid = await validateCfAccess(req, { teamDomain: cfTeamDomain, ... });
      if (!valid) return new Response(JSON.stringify({ error: "Unauthorized" }), { status: 401 });
    }
    
    return router(req);
  },
  websocket: {
    open(ws) { events.subscribe(ws.data.channel, ws); },
    close(ws) { events.unsubscribe(ws.data.channel, ws); },
    message(ws, msg) { /* handle pong */ },
  },
});
```

For event emission, add hooks at key points:
- In AlertSystem: after `sendMessage`, also call `events.emit("notifications", { type: "alert", data: payload })`
- In gate system: after gate creation, emit `gate_pending`; after resolution, emit `gate_resolved`
- In SessionRunner: after status changes, emit to `session:${sessionId}` channel

These hooks should be added via the DI pattern — pass `events` as an optional dep to subsystems.

## What NOT to Do

- Do not add environment variables for API configuration -- use Docker secrets consistently
- Do not break existing Telegram integration -- event emission is additive, not replacement
- Do not require Cloudflare Tunnel to be running for Docker build or tests
- Do not add rate limiting -- single user system, unnecessary complexity

## Recovery Guidance

- If wiring breaks existing tests, ensure all new deps are optional (events?, resolveGateFn?)
- If WebSocket upgrade fails, check Bun.serve docs for correct `server.upgrade()` usage
- If Docker build fails, verify no new packages were added that aren't in bun.lock
- Test the API server manually: `curl http://localhost:9999/health` and `curl http://localhost:9999/api/sessions`
