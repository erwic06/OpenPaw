# Task 2.2 -- Telegram Messaging Integration

## Objective

Integrate a Telegram bot that NanoClaw uses to send notifications and receive human responses.

## Context

Telegram is the primary HITL communication channel in Phase 2. NanoClaw sends task-ready notifications and HITL gate requests via Telegram, and receives approve/deny responses. The bot will later handle task intake commands (/research, /project, etc.) in Phase 6, but this task covers only the messaging infrastructure. Depends on 2.1 (container must be running).

## Requirements

- [ ] Telegram bot created via BotFather, token stored in secrets.env
- [ ] Bot connects to Telegram API on NanoClaw startup
- [ ] Send formatted messages to a configured chat ID
- [ ] Receive and parse incoming messages from the authorized user
- [ ] Message formatting supports the alert format from design doc Section 5
- [ ] Graceful reconnection on network interruption

## Interface Contract

**Inputs:** `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` from environment/secrets.

**Outputs:** Exported functions: `sendMessage(chatId, text, options?)`, `onMessage(handler)`.

**Side Effects:** Opens a long-polling connection to Telegram API.

## Technical Approach

Use the `grammy` library (lightweight, TypeScript-native Telegram bot framework). Long-polling mode (not webhooks) since NanoClaw runs behind NAT without a public URL in Phase 2. Export a clean interface that other modules (plan reader, HITL gates) can import.

## Files to Create/Modify

- `src/messaging/telegram.ts` -- Bot initialization, send/receive functions
- `src/messaging/index.ts` -- Messaging module exports
- `package.json` -- Add grammy dependency

## Acceptance Criteria

- [ ] Bot starts and connects to Telegram API (visible in container logs)
- [ ] `sendMessage` delivers a test message to the configured chat
- [ ] Incoming messages from the authorized user trigger the registered handler
- [ ] Messages from unauthorized users are ignored
- [ ] Bot reconnects after a simulated network interruption (stop/start polling)
- [ ] Alert format from design doc renders correctly in Telegram (bold, line breaks)

## What NOT to Do

- Do not implement command parsing (/research, /project, etc.) -- that's Phase 6
- Do not implement HITL gate logic -- that's task 2.5
- Do not set up webhooks -- use long-polling for Phase 2
- Do not hardcode the bot token or chat ID

## Recovery Guidance

- If bot can't connect: verify token is correct, check that BotFather shows the bot as active
- If messages don't deliver: verify chat ID is correct (use @userinfobot to find your chat ID)
- If grammy has issues: it's well-maintained but check for Bun compatibility; fall back to node-telegram-bot-api if needed
