# Task 5.3 -- Stuck Task Detection

## Objective

Detect tasks that fail 3 consecutive times, pause dispatch for that specific task, and send a `stuck_task` alert with full failure history per design doc Section 4 F3 (lines 786-788).

## Context

The design doc specifies: "If same task fails 3 times: pause dispatch for that task, notify human with full failure history, wait for human intervention (rewrite contract, adjust acceptance criteria, or take over interactively)."

Currently, SessionRunner will keep retrying a failed task every time the plan watcher detects it as `ready` (after recovery resets it). There is no limit on retries and no escalation. The sessions table already tracks `task_id` and `terminal_state`, so failure counts can be queried directly.

Depends on Task 5.1 (AlertSystem for `stuck_task` alert). Integrates into SessionRunner's post-task completion path.

## Requirements

- [ ] `getTaskFailureCount(db: Database, taskId: string): number` -- query `sessions` table for count of `FAILED` terminal states for a given `task_id`
- [ ] SessionRunner maintains a `stuckTasks: Set<string>` -- tasks that have hit the 3-failure threshold
- [ ] In `SessionRunner.runTask()`, after a task terminates with `finalStatus === "failed"`: call `getTaskFailureCount`. If >= 3, add `task.id` to `stuckTasks` set and send `stuck_task` alert via AlertSystem
- [ ] In `SessionRunner.drainQueue()`: skip tasks whose `id` is in `stuckTasks` (log skip, don't silently drop)
- [ ] `stuck_task` alert includes: taskId, taskTitle, failureCount, last error message
- [ ] `stuckTasks` set is not persisted -- it resets on NanoClaw restart. This is intentional: the human may have fixed the contract between restarts, so a fresh retry is appropriate
- [ ] Optional `alertSystem` field added to `RunnerDeps` for stuck task alerts (null-safe, skips alert if not provided)

## Interface Contract

**Inputs:** `Database` (sessions table for failure count query), task completion events.

**Outputs:** `getTaskFailureCount` exported from `src/db/index.ts`. Stuck task behavior in SessionRunner.

**Side Effects:** Sends `stuck_task` alert via AlertSystem. Skips dispatch for stuck tasks.

## Technical Approach

```typescript
// In src/db/index.ts
export function getTaskFailureCount(db: Database, taskId: string): number {
  const result = db.prepare(
    "SELECT COUNT(*) as count FROM sessions WHERE task_id = ? AND terminal_state = 'FAILED'"
  ).get(taskId) as { count: number };
  return result.count;
}
```

In `SessionRunner.runTask()`, after the plan update at line 207-212:
```typescript
if (finalStatus === "failed") {
  const failCount = getTaskFailureCount(this.deps.db, task.id);
  if (failCount >= 3) {
    this.stuckTasks.add(task.id);
    if (this.deps.alertSystem) {
      await this.deps.alertSystem.send({
        type: "stuck_task",
        taskId: task.id,
        taskTitle: task.title,
        failureCount: failCount,
        lastError: statusNotes ?? "unknown",
      });
    }
  }
}
```

In `SessionRunner.drainQueue()`:
```typescript
while (this.queue.length > 0) {
  // ... budget check ...
  const task = this.queue.shift()!;
  if (this.stuckTasks.has(task.id)) {
    console.log(`[runner] skipping stuck task ${task.id} (3+ failures)`);
    continue;
  }
  // ...
}
```

## What NOT to Do

- Do not persist stuckTasks to disk or database -- restart is the human's reset mechanism
- Do not implement auto-recovery from stuck state (the human decides: rewrite contract, adjust criteria, or take over)
- Do not apply stuck task detection to research tasks (ResearchRunner has its own flow)
- Do not add a configurable failure threshold -- hardcode 3 per the design doc

## Recovery Guidance

- If the stuckTasks check breaks drainQueue, it's safe to remove the check -- the Set starts empty so existing tests won't be affected.
- If getTaskFailureCount returns wrong counts, check whether the sessions table uses `terminal_state = 'FAILED'` (uppercase, matching the monitor/recovery convention) or `'failed'` (lowercase). Use the actual value stored.
