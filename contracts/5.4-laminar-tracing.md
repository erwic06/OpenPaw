# Task 5.4 -- Laminar Tracing Integration

## Objective

Integrate the Laminar JS SDK so every headless session (coding and research) produces a Laminar trace, with secret sanitization in trace metadata, per design doc Section 5 (lines 845-873).

## Context

The design doc specifies Laminar for observability: each headless session is a Laminar "trace" containing session ID, agent type, task ID, model, provider, and terminal state. Laminar's SDK auto-instruments downstream HTTP calls (Anthropic, OpenAI, Google APIs) via OpenTelemetry, so individual inference calls appear as child spans automatically.

Interactive Claude Code sessions are NOT traced (human-supervised, logged by Claude Code itself).

No dependencies on other Phase 5 tasks. Can be built in parallel with 5.1, 5.2, 5.3, 5.5.

## Requirements

- [ ] Install `@lmnr-ai/lmnr` (exact pinned version per dependency policy)
- [ ] `TracingDeps` interface: `{ laminarApiKey?: string; secretValues: Set<string> }`
- [ ] `initTracing(deps: TracingDeps): void` -- calls `Laminar.initialize()` if key present; sets no-op flag if absent
- [ ] `traceSession<T>(name: string, metadata: Record<string, string>, fn: () => Promise<T>): Promise<T>` -- wraps `observe()` if initialized, else calls `fn()` directly
- [ ] `shutdownTracing(): Promise<void>` -- calls `Laminar.shutdown()` if initialized, no-op otherwise
- [ ] `scrubSecrets(text: string, secrets: Set<string>): string` -- replaces any substring matching a secret value with `[REDACTED]`
- [ ] Wrap `SessionRunner.executeSession()` in a traceSession call with metadata `{ agent: "coder", taskId, model }`
- [ ] Wrap `ResearchRunner.executeDefaultResearch()` in a traceSession call with metadata `{ agent: "researcher", taskId, model, depth }`
- [ ] Trace metadata is scrubbed before passing to Laminar (no secret values in traces)
- [ ] Graceful degradation: if `laminar_api_key` missing, all tracing functions are no-ops; never blocks or fails session execution
- [ ] Tracing errors caught and logged (never propagated to caller)

## Interface Contract

**Inputs:** `laminar_api_key` secret (optional), `Set<string>` of all secret values for scrubbing.

**Outputs:** `initTracing`, `traceSession`, `shutdownTracing`, `scrubSecrets` exported from `src/tracing/index.ts`.

**Side Effects:** Sends trace data to Laminar's API when initialized. No side effects when key is absent.

## Technical Approach

```typescript
// src/tracing/index.ts
let initialized = false;

export function initTracing(deps: TracingDeps): void {
  if (!deps.laminarApiKey) {
    console.log("[tracing] no laminar_api_key — tracing disabled");
    return;
  }
  try {
    const { Laminar } = require("@lmnr-ai/lmnr");
    Laminar.initialize({ projectApiKey: deps.laminarApiKey });
    initialized = true;
    console.log("[tracing] Laminar initialized");
  } catch (err) {
    console.error("[tracing] failed to initialize Laminar:", err);
  }
}

export async function traceSession<T>(
  name: string,
  metadata: Record<string, string>,
  fn: () => Promise<T>,
): Promise<T> {
  if (!initialized) return fn();
  try {
    const { observe } = require("@lmnr-ai/lmnr");
    return await observe({ name, ...metadata }, fn);
  } catch {
    return fn();
  }
}

// src/tracing/sanitize.ts
export function scrubSecrets(text: string, secrets: Set<string>): string {
  let result = text;
  for (const secret of secrets) {
    if (secret.length >= 4 && result.includes(secret)) {
      result = result.replaceAll(secret, "[REDACTED]");
    }
  }
  return result;
}
```

```typescript
// In SessionRunner.executeSession()

```

## What NOT to Do

- Do not add child spans manually for individual adapter calls -- Laminar auto-instruments HTTP calls
- Do not trace interactive Claude Code sessions
- Do not build a custom trace UI -- use Laminar's dashboard
- Do not store traces locally -- Laminar's cloud handles storage
- Do not block session execution on tracing failures
- Do not add laminar_api_key to docker-compose.yml in this task -- that is task 5.6

## Recovery Guidance

- If `@lmnr-ai/lmnr` has compatibility issues with Bun, check if there's a Bun-specific import path or if the package needs Node.js compat mode. The package uses OpenTelemetry under the hood which may need `--conditions=node` in Bun.
- If the package has too many transitive deps, consider wrapping only `observe()` as an optional dynamic import.
- The no-op fallback means the system works identically without the package.
