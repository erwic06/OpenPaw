# Task 8.5 -- Cron Scheduler

## Objective

Implement a pure-function cron expression evaluator and the `CronScheduler` class that checks enabled cron-type agents every 60 seconds and triggers them at the correct times.

## Context

Scheduled agents use standard 5-field cron expressions. NanoClaw evaluates them internally — no external cron daemon or library needed. The scheduler runs as a `setInterval` loop inside the NanoClaw process. It checks all enabled cron-type definitions and triggers any whose expression matches the current time.

Depends on 8.1 (types) and 8.3 (loader for definition updates). Blocks 8.6 (event triggers share the `triggerAgent` callback pattern).

## Requirements

### Cron Evaluator (pure functions)
- [ ] `matchesCron(expression: string, date: Date): boolean` -- check if date matches cron expression
- [ ] `nextCronTime(expression: string, after: Date): Date` -- compute next matching time
- [ ] 5-field format: minute (0-59), hour (0-23), day-of-month (1-31), month (1-12), day-of-week (0-6, 0=Sunday)
- [ ] Support `*` (any), specific values (`5`), lists (`1,3,5`), ranges (`1-5`), step values (`*/5`)
- [ ] No timezone handling -- all times in system local time

### CronScheduler Class
- [ ] `CronScheduler` class with `SchedulerDeps` DI interface
- [ ] `SchedulerDeps`: `{ db: Database; triggerAgent: (def: AgentDefinition, triggeredBy: string, detail: string) => Promise<void>; now?: () => Date }`
- [ ] `start(): void` -- begins the 60-second interval loop
- [ ] `stop(): void` -- clears the interval
- [ ] `updateSchedule(definitions: AgentDefinition[]): void` -- called by file watcher when definitions change
- [ ] `triggerManual(agentId: string): Promise<void>` -- on-demand execution for "Run now" button
- [ ] Each tick: get enabled cron-type definitions from DB, check `matchesCron`, trigger matches
- [ ] Idempotent: check `last_run_at` to avoid double-firing (skip if `last_run_at` is within same cron minute)
- [ ] Update `last_run_at` and `next_run_at` on each trigger
- [ ] `now()` DI for testability with fake timers

## Interface Contract

**Inputs:** Agent definitions from DB, current time.

**Outputs:** `CronScheduler` class, `matchesCron`/`nextCronTime` functions.

**Side Effects:** Calls `triggerAgent` callback on schedule matches. Updates `last_run_at`/`next_run_at` in DB.

## Technical Approach

```typescript
// src/fleet/cron.ts
export function matchesCron(expression: string, date: Date): boolean {
  const [minExpr, hourExpr, domExpr, monExpr, dowExpr] = expression.trim().split(/\s+/);
  return (
    matchField(minExpr, date.getMinutes(), 0, 59) &&
    matchField(hourExpr, date.getHours(), 0, 23) &&
    matchField(domExpr, date.getDate(), 1, 31) &&
    matchField(monExpr, date.getMonth() + 1, 1, 12) &&
    matchField(dowExpr, date.getDay(), 0, 6)
  );
}

function matchField(expr: string, value: number, min: number, max: number): boolean {
  if (expr === "*") return true;
  // Handle lists (1,3,5), ranges (1-5), steps (*/5, 1-10/2)
}
```

## What NOT to Do

- Do not use a cron parsing library (node-cron, croner, etc.) -- the evaluator is ~100 lines
- Do not add timezone support -- system local time is fine for single-user system
- Do not add second-level precision -- minute granularity is sufficient
- Do not implement event triggers -- that is task 8.6
