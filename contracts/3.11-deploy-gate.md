# Task 3.11 -- Deploy Gate Wiring

## Objective

Wire Gate 2 (Deploy Approval) into the session runner: when a completed task is tagged for deployment, assemble context (git diff, test results, Cubic review summary) and request human approval via the existing HITL gate infrastructure.

## Context

The deploy gate (Gate 2) is already defined in the gate infrastructure (Task 2.5) with a 24-hour timeout. This task adds the trigger logic: detect deploy-tagged tasks, assemble the context a human needs to make an approval decision, and call `requestApproval`. The plan parser needs to be extended to recognize deploy tags on tasks.

Depends on Task 3.9 (session runner).

## Requirements

- [ ] Extend plan parser to detect `deploy: production` or `deploy: staging` tags on tasks
- [ ] Add `deploy` optional field to `Task` type in `src/plan/types.ts`
- [ ] After a task completes successfully in `SessionRunner.runTask`, check for deploy tag
- [ ] If deploy-tagged: assemble context:
  - Git diff (changes since last deploy or since branch creation)
  - Test results summary from session output
  - Cubic review summary (from Task 3.10) if available
- [ ] Call `requestApproval({ gateType: 'deploy', taskId, sessionId, contextSummary })`
- [ ] On approved: log approval (actual Vercel deploy integration is out of scope for Phase 3)
- [ ] On denied/timeout: mark task as BLOCKED with note
- [ ] Unit tests

## Interface Contract

**Inputs:** Completed task with deploy tag. Session output with test results. Cubic review (optional).

**Outputs:** Gate decision (approved/denied/timeout).

**Side Effects:** Sends deploy approval request via Telegram, writes gate record to SQLite.

## Technical Approach

In the plan markdown, tasks may have a deploy tag in their metadata:
```
- **Deploy:** production
```

The parser scans for this pattern and populates `task.deploy`. The session runner checks this field after successful completion and triggers the gate flow.

Context assembly concatenates: git diff summary (truncated to fit Telegram limits), test pass/fail counts, Cubic findings (if any).

## Files to Create/Modify

- `src/plan/types.ts` -- add `deploy?: 'production' | 'staging'` to `Task`
- `src/plan/parser.ts` -- parse deploy tag from task metadata
- `src/agents/runner.ts` -- add deploy gate logic after task completion
- `tests/deploy-gate.test.ts` -- unit tests

## Acceptance Criteria

- [ ] `bun run typecheck` passes
- [ ] `bun test` passes -- all new and existing tests pass
- [ ] Parser correctly extracts `deploy: production` and `deploy: staging` tags
- [ ] Deploy gate is triggered for deploy-tagged tasks on successful completion
- [ ] Context summary includes git diff, test results, and Cubic review (if available)
- [ ] Gate approval/denial is handled correctly
- [ ] Non-deploy tasks do not trigger the gate

## What NOT to Do

- Do not implement actual Vercel deployment -- just the gate trigger and approval flow
- Do not auto-approve staging deploys yet -- all manual in Phase 3
- Do not modify the gate infrastructure itself -- use it as-is from Task 2.5

## Recovery Guidance

- If parser doesn't detect deploy tags: check the exact format in `implementation_plan.md`
- If context is too long for Telegram: truncate the git diff and include a "full diff available via git" note
- If gate timeout fires unexpectedly: verify the 24h timeout in `GATE_CONFIGS`
