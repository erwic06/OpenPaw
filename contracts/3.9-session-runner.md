# Task 3.9 -- Session Runner + Orchestrator Wiring

## Objective

Implement the session runner that orchestrates the full headless task lifecycle: read contract, create sandbox, configure adapter, run session, update plan, commit, notify. Wire it into `src/index.ts` so NanoClaw automatically dispatches sessions for ready tasks.

## Context

This is the integration task that connects all Phase 3 components. The plan reader (Task 2.3) detects ready tasks. The session runner picks them up, creates a Daytona sandbox (Task 3.4), configures the LLMAdapter (Task 3.5) with MCP tools (Task 3.6), system prompt (Task 3.2), fallback routing (Task 3.7), and monitoring (Task 3.8), runs the session, and handles results.

Also includes a plan-writer utility: Phase 2's plan reader only reads `implementation_plan.md`. Phase 3 needs to write task status back after session completion.

Depends on Tasks 3.2, 3.6, 3.7, 3.8.

## Requirements

- [ ] `SessionRunner.runTask(task)` orchestrates the full lifecycle:
  1. Read the task's contract file
  2. Read `agents/coder/system_prompt.md`
  3. Create Daytona sandbox via `SandboxManager`
  4. Create MCP tool server via `createDaytonaToolServer`
  5. Configure `LLMAdapter` with model tier, MCP server, fallback routing
  6. Start session monitoring
  7. Execute session via `FallbackRouter` + `LLMAdapter`
  8. On completion: update `implementation_plan.md` (task status), commit changes via git
  9. Destroy sandbox
  10. Send Telegram notification of result
  11. Log cost summary
- [ ] Plan writer: `updateTaskStatus(planPath, taskId, newStatus, notes?)` -- find task in plan, update status line
- [ ] Wire into `src/index.ts`: initialize all subsystems, connect plan reader `onReady` to runner
- [ ] Sequential dispatch: only one headless session at a time (design doc: deferred concurrent dispatch)
- [ ] `src/index.ts` initializes: database, Telegram bot, gate system, session runner, plan watcher

## Interface Contract

**Inputs:** `Task` from plan reader (task ID, contract path, metadata).

**Outputs:** Session result (complete/failed/blocked). Side effects: plan updated, changes committed, notifications sent.

**Side Effects:** Runs headless session, modifies `implementation_plan.md`, creates git commits, sends Telegram messages, writes to SQLite.

## Technical Approach

The `SessionRunner` coordinates all components:

```typescript
class SessionRunner {
  async runTask(task: Task): Promise<void> {
    const contract = await readFile(task.contractPath);
    const systemPrompt = await readFile('agents/coder/system_prompt.md');
    const sandbox = await sandboxManager.createSandbox(sessionId, repoUrl, branch);
    const toolServer = createDaytonaToolServer(sandbox);
    // ... configure and run
  }
}
```

Plan writer uses simple string replacement: find the task by ID pattern (`### {taskId}`), locate the `**Status:**` line, replace the status value. This is fragile but matches the existing plan format exactly.

`src/index.ts` changes: after initializing Telegram bot, init database, init gates, create runner, start plan watcher with `onReady` callback that feeds tasks to the runner.

## What NOT to Do

- Do not implement parallel/concurrent dispatch -- sequential only per design doc
- Do not implement the deploy gate trigger here -- that's Task 3.11
- Do not implement restart recovery here -- that's Task 3.12
- Do not add new packages

## Recovery Guidance

- If plan writer corrupts the file: add a backup step (copy plan before modifying), or use a more robust parsing approach
- If sandbox creation fails mid-flow: ensure cleanup happens in a finally block
- If index.ts becomes too complex: extract initialization into a separate `src/bootstrap.ts`
