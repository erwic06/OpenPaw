# Task 5.1 -- Structured Alert System

## Objective

Replace ad-hoc `sendAlert("string literal")` calls across the codebase with typed alert payloads, consistent HTML formatting matching design doc Section 5, and support for a dedicated alerts Telegram channel separate from HITL gates.

## Context

Currently, alerts are sent as raw strings via a `sendAlert: (message: string) => Promise<void>` callback passed through DI deps in `SessionRunner`, `ResearchRunner`, `SessionMonitor`, `recoverOrphanedSessions`, and `executeWithFallback`. Each call site formats its own message string, leading to inconsistent formatting and no type safety.

The design doc (Section 5, lines 912-943) specifies 7 alert types with a consistent format. Section 10 (line 1673) specifies alerts should go to a separate Telegram chat from HITL gates. This task creates the alert type system and formatter as the foundation for budget controls (5.2) and stuck task detection (5.3).

No dependencies. Blocks tasks 5.2 and 5.3.

## Requirements

- [ ] `AlertType` union: `"session_failed" | "session_blocked" | "budget_warning" | "budget_hard_stop" | "stuck_task" | "orchestrator_restart" | "fallback_activated"`
- [ ] `AlertPayload` discriminated union with per-type fields:
  - `session_failed`: taskId, taskTitle, agent, model, durationMs, error, costUsd
  - `session_blocked`: taskId, taskTitle, agent, model, reason
  - `budget_warning`: dailySpendUsd, dailyLimitUsd, thresholdPct
  - `budget_hard_stop`: dailySpendUsd, dailyLimitUsd
  - `stuck_task`: taskId, taskTitle, failureCount, lastError
  - `orchestrator_restart`: orphanedCount, taskIds
  - `fallback_activated`: primaryModel, fallbackModel, taskId?, error
- [ ] `formatAlertMessage(payload: AlertPayload): string` produces HTML matching the design doc format (emoji header, bold field labels, footer with action taken)
- [ ] `AlertSystem` class with `AlertDeps` interface: `{ sendMessage: (chatId, text, options?) => Promise<void>; alertsChatId?: string; fallbackChatId: string }`
- [ ] `AlertSystem.send(payload: AlertPayload): Promise<void>` formats and sends to `alertsChatId` if set, else `fallbackChatId`
- [ ] Existing `sendAlert` DI signature unchanged -- `AlertSystem` is composed *around* it, not replacing the DI interface
- [ ] Barrel export from `src/alerts/index.ts`

## Interface Contract

**Inputs:** `AlertPayload` (typed alert data).

**Outputs:** `AlertSystem` class, `AlertPayload` types, `formatAlertMessage` function. All exported from `src/alerts/index.ts`.

**Side Effects:** Sends formatted HTML messages to Telegram via the provided `sendMessage` function.

## Technical Approach

```typescript
// src/alerts/types.ts
export type AlertType =
  | "session_failed"
  | "session_blocked"
  | "budget_warning"
  | "budget_hard_stop"
  | "stuck_task"
  | "orchestrator_restart"
  | "fallback_activated";

export type AlertPayload =
  | { type: "session_failed"; taskId: string; taskTitle: string; agent: string; model: string; durationMs: number; error: string; costUsd: number }
  | { type: "session_blocked"; taskId: string; taskTitle: string; agent: string; model: string; reason: string }
  // ... etc

// src/alerts/formatter.ts
// Maps AlertType → { emoji, title, fields[], footer }
// Reuses the escapeHtml pattern from src/messaging/telegram.ts

// src/alerts/index.ts
export class AlertSystem {
  constructor(private deps: AlertDeps) {}
  async send(payload: AlertPayload): Promise<void> {
    const html = formatAlertMessage(payload);
    const chatId = this.deps.alertsChatId ?? this.deps.fallbackChatId;
    await this.deps.sendMessage(chatId, html, { parseMode: "HTML" });
  }
}
```

The existing `sendAlert` calls in runner/monitor/recovery/fallback will NOT be changed in this task. Those call sites will be migrated to typed alerts incrementally in tasks 5.2, 5.3, and 5.6 as each module is touched. This task only creates the alert infrastructure and exports.
