# Task 5.5 -- SQL Cost Views

## Objective

Add three queryable SQL views for cost reporting to the database schema per design doc Section 5 (lines 894-907), and create corresponding TypeScript query wrappers.

## Context

Cost tracking data already exists in the `cost_log` and `sessions` tables. The design doc specifies three views for operational cost analysis: daily spend by service, monthly spend by agent type, and most expensive sessions. These views make the data queryable without writing ad-hoc SQL each time.

The existing `initDatabase` function in `src/db/index.ts` already splits `schema.sql` by semicolons and executes each statement, so adding `CREATE VIEW IF NOT EXISTS` statements to the schema file will work with the existing initialization flow.

No dependencies on other Phase 5 tasks. Can be built in parallel with 5.1, 5.4.

## Requirements

- [ ] Three SQL views added to `src/db/schema.sql`:
  1. `daily_spend_by_service` -- `date(logged_at) as day, service, SUM(amount_usd) as total` from `cost_log`, grouped by day and service
  2. `monthly_spend_by_agent` -- `s.agent, SUM(c.amount_usd) as total` joining `cost_log c` with `sessions s` on `c.session_id = s.id`, filtered to current month, grouped by agent
  3. `most_expensive_sessions` -- `s.id, s.agent, s.task_id, s.model, s.started_at, SUM(c.amount_usd) as total_cost` joining `cost_log c` with `sessions s`, grouped by `s.id`, ordered by `total_cost DESC`
- [ ] TypeScript query wrappers in `src/db/index.ts`:
  - `getDailySpendByService(db, date?: string): Array<{ day: string; service: string; total: number }>` -- queries view, optionally filtered by date
  - `getMonthlySpendByAgent(db): Array<{ agent: string; total: number }>` -- queries view for current month
  - `getMostExpensiveSessions(db, limit?: number): Array<{ id: string; agent: string; task_id: string | null; model: string; started_at: string; total_cost: number }>` -- queries view with configurable limit (default 10)
- [ ] Views use `CREATE VIEW IF NOT EXISTS` for idempotent initialization
- [ ] Views correctly handle NULL session_id in cost_log (left join or inner join as appropriate)

## Interface Contract

**Inputs:** Existing `cost_log` and `sessions` tables.

**Outputs:** Three SQL views, three TypeScript query functions exported from `src/db/index.ts`.

**Side Effects:** Views created in SQLite during database initialization.

## Technical Approach

Add to `src/db/schema.sql` after the table definitions:

```sql
CREATE VIEW IF NOT EXISTS daily_spend_by_service AS
SELECT date(logged_at) AS day, service, SUM(amount_usd) AS total
FROM cost_log
GROUP BY day, service;

CREATE VIEW IF NOT EXISTS monthly_spend_by_agent AS
SELECT s.agent, SUM(c.amount_usd) AS total
FROM cost_log c
JOIN sessions s ON c.session_id = s.id
WHERE c.logged_at >= date('now', 'start of month')
GROUP BY s.agent;

CREATE VIEW IF NOT EXISTS most_expensive_sessions AS
SELECT s.id, s.agent, s.task_id, s.model, s.started_at, SUM(c.amount_usd) AS total_cost
FROM cost_log c
JOIN sessions s ON c.session_id = s.id
GROUP BY s.id
ORDER BY total_cost DESC;
```

TypeScript wrappers are thin queries against these views:

```typescript
export function getDailySpendByService(
  db: Database,
  date?: string,
): Array<{ day: string; service: string; total: number }> {
  if (date) {
    return db.prepare("SELECT * FROM daily_spend_by_service WHERE day = date(?)").all(date) as any;
  }
  return db.prepare("SELECT * FROM daily_spend_by_service ORDER BY day DESC").all() as any;
}
```

## What NOT to Do

- Do not build a cost dashboard UI -- use SQLite queries and these views
- Do not modify the existing `getDailySpend` function -- it serves a different purpose (single total for budget enforcement)
- Do not add indexes for the views -- at <100 sessions/month the tables are tiny
- Do not add views for data that doesn't exist yet (e.g., agent_definitions from Phase 8)

## Recovery Guidance

- If `CREATE VIEW IF NOT EXISTS` fails on older SQLite, check Bun's embedded SQLite version. The `IF NOT EXISTS` clause for views was added in SQLite 3.7.11 (2012), so this should not be an issue.
- If the `monthly_spend_by_agent` view returns stale results, it's because `date('now')` in the view evaluates at query time, which is correct behavior.
