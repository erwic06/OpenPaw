# Task 2.1 -- NanoClaw Docker Container Setup

## Objective

Create the Docker infrastructure for NanoClaw so it runs as a containerized daemon on the Mac Mini.

## Context

NanoClaw is the orchestration daemon for OpenPaw. It runs in Docker with mounted volumes for the git repo, secrets, and SQLite data directory. All subsequent Phase 2 tasks build on this container. See design doc Section 2 (Container Strategy).

## Requirements

- [ ] Multi-stage Dockerfile based on Bun base image, pinned by digest (not tag)
- [ ] Builder stage: `bun install --ignore-scripts --frozen-lockfile`
- [ ] Runtime stage: non-root user (`nanoclaw`, uid 1000), only production artifacts copied from builder
- [ ] docker-compose.yml with:
  - Volume mounts: git repo (read-write), SQLite data dir (read-write)
  - Docker Compose secrets for API keys (mounted from `~/.nanoclaw/secrets/`, one file per key)
  - `read_only: true` root filesystem with writable tmpfs for `/tmp`
  - `no-new-privileges: true` in security_opt
  - `restart: unless-stopped`
- [ ] .dockerignore excluding node_modules, .git, .env*, *.md, test/
- [ ] Container runs as a long-lived daemon (not one-shot)
- [ ] Health check endpoint or process monitor
- [ ] `bun.lock` committed to git
- [ ] `bun audit` runs during build (logs warnings, does not block build)

## Interface Contract

**Inputs:** API key files in `~/.nanoclaw/secrets/` (mounted as Docker Compose secrets, read-only), git repo directory (mounted read-write), SQLite data directory (mounted read-write).

**Outputs:** Running NanoClaw container accessible on the host network. Secrets loader module that reads `/run/secrets/*` into an in-memory Map.

**Side Effects:** Creates Docker container, volumes, and secrets on the Mac Mini.

## Technical Approach

Use Bun as the runtime (faster startup, native TypeScript). Multi-stage Dockerfile:

Stage 1 (builder): `FROM oven/bun@sha256:<digest>`. Copy package.json and bun.lock. Run `bun install --ignore-scripts --frozen-lockfile --production`. This ensures reproducible builds with no lifecycle script execution.

Stage 2 (runtime): `FROM oven/bun@sha256:<digest>`. Create nanoclaw user (uid 1000). Copy node_modules from builder. Copy application source. Set `USER nanoclaw`. Entrypoint runs the NanoClaw daemon process.

docker-compose.yml: Define secrets from `~/.nanoclaw/secrets/` files. Mount git repo and SQLite data dir as volumes. Set `read_only: true` on the container with tmpfs mount for `/tmp`. Set `security_opt: no-new-privileges:true`. Use `restart: unless-stopped`.

The entry point `src/index.ts` imports a secrets loader (`src/secrets.ts`) that reads `/run/secrets/*` at startup into an in-memory Map (never process.env).

## Files to Create/Modify

- `Dockerfile` -- NanoClaw container definition
- `docker-compose.yml` -- Service definition with volume mounts
- `.dockerignore` -- Build context exclusions
- `src/index.ts` -- Minimal entry point (starts the daemon, loads secrets, logs startup)
- `src/secrets.ts` -- Secrets loader (reads /run/secrets/*, returns Map<string, string>)
- `package.json` -- Project manifest with Bun/Node config (exact version pins only)
- `tsconfig.json` -- TypeScript configuration
- `bun.lock` -- Lock file (generated by bun install, committed to git)

## Acceptance Criteria

- [ ] `docker compose build` succeeds without errors
- [ ] `docker compose up -d` starts the container and it stays running for 30+ seconds
- [ ] Container logs show NanoClaw startup message
- [ ] Git repo directory is accessible inside the container at the mounted path
- [ ] Secrets are readable at `/run/secrets/<name>` inside the container
- [ ] Secrets do NOT appear in `docker inspect` output
- [ ] `docker compose exec nanoclaw whoami` returns `nanoclaw` (not root)
- [ ] `docker compose exec nanoclaw touch /testfile` fails (read-only root filesystem)
- [ ] Container restarts automatically if the process exits
- [ ] `bun.lock` exists and is committed to git

## What NOT to Do

- Do not implement any NanoClaw business logic beyond a startup message -- that's for later tasks
- Do not install LLM provider SDKs yet
- Do not configure networking beyond basic host access
- Do not create the secret files (they're user-managed, outside the repo at ~/.nanoclaw/secrets/)
- Do not pass secrets as environment variables -- use Docker Compose secrets exclusively

## Recovery Guidance

- If Docker build fails: check base image availability, ensure Bun image tag exists
- If volumes don't mount: verify paths exist on host, check Docker Desktop permissions on macOS
- If container exits immediately: check entry point script, ensure it has a long-running process (not just a script that exits)
