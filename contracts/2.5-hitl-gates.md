# Task 2.5 -- HITL Gate Infrastructure

## Objective

Implement the human-in-the-loop gate system that sends approval requests via Telegram and processes human responses.

## Context

HITL gates are the safety mechanism for high-stakes actions. In Phase 2, gates are sent via Telegram and resolved via Telegram replies. The gate infrastructure creates gate records in SQLite, sends formatted requests via the Telegram module, and processes approve/deny responses. The design doc defines 6 gate types; all 6 must be supported by the infrastructure, though Gate 6 (external_communication) won't fire until external service agents are built in Phase 8. Depends on 2.2 (Telegram messaging) and 2.4 (SQLite database).

## Requirements

- [ ] Support all 6 gate types from design doc Section 3: plan, deploy, research, spec_change, spend, external_communication
- [ ] Gate 6 (external_communication) won't fire in Phase 2 but the type must be supported in the gate infrastructure
- [ ] Create a gate record in SQLite when a gate is triggered
- [ ] Send a formatted approval request via Telegram with context summary
- [ ] Process human responses: "approve"/"yes" = approved, "deny"/"no" = rejected, other text = feedback
- [ ] Update gate record with decision and timestamp
- [ ] Support timeout behavior per gate type (some timeout, some wait indefinitely)
- [ ] Return the decision to the calling code (async -- waits for human response)

## Interface Contract

**Inputs:** Gate type, task ID, session ID, context summary.

**Outputs:**
- `requestApproval(gate): Promise<GateDecision>` -- send gate, wait for response, return decision
- `getPendingGates(): Gate[]` -- list unresolved gates

**Side Effects:** Sends Telegram messages, writes to SQLite hitl_gates table.

## Technical Approach

When `requestApproval` is called: insert a `hitl_gates` row with `decision: null`, send a Telegram message with the gate details, register a message handler that matches the gate ID. When the human replies, update the row and resolve the promise. Use a Map of pending gate promises keyed by gate ID.

For timeout: use `Promise.race` with a timeout that resolves to `{ decision: 'timeout' }` for gates that have a timeout (deploy: 24h). Gates without timeout (plan, spec_change) wait indefinitely.

## Files to Create/Modify

- `src/gates/index.ts` -- Gate lifecycle: create, send, await, resolve
- `src/gates/types.ts` -- Gate types, decision types, gate config per type
- `src/gates/formatter.ts` -- Format gate requests for Telegram display

## Acceptance Criteria

- [ ] `requestApproval` sends a formatted message to Telegram with gate details
- [ ] Replying "approve" resolves the gate as approved and updates SQLite
- [ ] Replying "deny" resolves the gate as rejected and updates SQLite
- [ ] Replying with other text attaches it as feedback on the gate
- [ ] Gate record in SQLite has correct timestamps for requested_at and decided_at
- [ ] `getPendingGates` returns only unresolved gates
- [ ] Timeout fires for gate types that have timeouts (deploy: 24h)
- [ ] Gate type `external_communication` is accepted by `requestApproval` and creates a valid gate record

## What NOT to Do

- Do not implement the business logic that triggers gates (plan approval, deploy approval, etc.) -- just the gate infrastructure
- Do not implement auto-approve rules yet -- all gates are manual in Phase 2
- Do not implement the web dashboard gate UI -- Telegram only for now
- Do not send gates to multiple channels -- single Telegram chat for Phase 2

## Recovery Guidance

- If gate responses aren't matching: check that the reply matching logic correctly associates a Telegram reply with the right pending gate
- If promises leak: ensure timeouts clean up the pending gate map; use a WeakMap or explicit cleanup
- If Telegram formatting breaks: simplify the message format; Telegram has limited markdown support (use MarkdownV2 or HTML mode)
