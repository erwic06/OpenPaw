# Task 3.7 -- Fallback Routing

## Objective

Implement model fallback routing: when the primary Anthropic model fails after retries, switch to the corresponding OpenAI fallback model per the model roster. Includes exponential backoff retries, Telegram notification on fallback, and an OpenAI adapter for the fallback path.

## Context

The design doc (Section 4, F1) specifies: retry with exponential backoff (30s, 60s, 120s, 3 attempts), then switch to fallback model. The model roster maps each tier to a fallback: Opus 4.6 -> GPT-5.4 High, Sonnet 4.6 -> GPT-5.4 Medium. The Claude Agent SDK only supports Anthropic models, so the OpenAI fallback path needs a separate adapter that manages the tool-call loop directly using the OpenAI SDK.

Depends on Task 3.5 (LLMAdapter).

## Requirements

- [ ] `FallbackRouter` wraps session execution with retry + fallback logic
- [ ] On rate limit / quota / outage errors from Anthropic: retry with exponential backoff (30s, 60s, 120s)
- [ ] After 3 failed retries: switch to fallback model per roster tier
- [ ] Send Telegram alert on fallback activation: "Fallback activated: {primary} -> {fallback}"
- [ ] If fallback also fails after retries: terminate session as BLOCKED
- [ ] `OpenAIAdapter` implements tool-call loop using OpenAI SDK for GPT-5.4 models
- [ ] OpenAI adapter accepts the same MCP tools and system prompt as the primary path
- [ ] DI pattern for testability
- [ ] Unit tests covering: successful retry, fallback activation, all-providers-exhausted

## Interface Contract

**Inputs:** Primary `ModelConfig`, fallback `ModelConfig`, session execution function.

**Outputs:** Session result (from whichever provider succeeded) or BLOCKED status.

**Side Effects:** Sends Telegram alerts. May switch API provider mid-session. Logs fallback events.

## Technical Approach

`FallbackRouter` wraps the execution path:

```typescript
async function executeWithFallback(
  roster: ModelConfig,
  executePrimary: () => Promise<AgentOutput>,
  executeFallback: () => Promise<AgentOutput>,
  deps: FallbackDeps
): Promise<AgentOutput> {
  // Try primary with retries
  // On exhaustion, notify and try fallback
  // On fallback exhaustion, return BLOCKED
}
```

The `OpenAIAdapter` uses the `openai` SDK to run a tool-call loop: send messages, receive tool calls, execute tools against the same MCP tool handlers, send results back. This is more manual than the Agent SDK path but necessary for cross-provider fallback.

Error detection: check for HTTP 429, quota-exceeded errors, connection timeouts from the Agent SDK's error messages or thrown exceptions.

- Do not implement automatic provider health checks -- react to errors only
- Do not add Google/Gemini as a fallback -- that's a future consideration

## Recovery Guidance

- If OpenAI tool-call loop is complex: simplify by supporting only the tools the Coder actually uses (file, shell, git) rather than generic MCP passthrough
- If rate limit detection is unreliable: add logging for all error types to identify the exact error shape
- If fallback timing is wrong: verify `Bun.sleep()` durations
