# Task 3.7 -- Fallback Routing

## Objective

Implement model fallback routing: when the primary model fails after retries, switch to the corresponding fallback model per the model roster. Includes exponential backoff retries, Telegram notification on fallback, and cross-provider fallback via CodexAdapter.

## Context

The design doc (Section 4, F1) specifies: retry with exponential backoff (30s, 60s, 120s, 3 attempts), then switch to fallback model. The model roster maps each tier to a fallback: Opus 4.6 -> GPT-5.4, Sonnet 4.6 -> GPT-5.4. The Claude Agent SDK only supports Anthropic models, so the OpenAI fallback path uses the CodexAdapter (via @openai/codex-sdk) which manages its own session lifecycle.

Depends on Task 3.5 (LLMAdapter).

## Requirements

- [ ] `FallbackRouter` wraps session execution with retry + fallback logic
- [ ] On rate limit / quota / outage errors from Anthropic: retry with exponential backoff (30s, 60s, 120s)
- [ ] After 3 failed retries: switch to fallback model per roster tier
- [ ] Send Telegram alert on fallback activation: "Fallback activated: {primary} -> {fallback}"
- [ ] If fallback also fails after retries: terminate session as BLOCKED
- [ ] `CodexAdapter` implements Codex sessions via `@openai/codex-sdk` for GPT-5.4 fallback
- [ ] CodexAdapter uses native tools (file/shell/git) — no MCP tools needed
- [ ] DI pattern for testability
- [ ] Unit tests covering: successful retry, fallback activation, all-providers-exhausted

## Interface Contract

**Inputs:** Primary `ModelConfig`, fallback `ModelConfig`, session execution function.

**Outputs:** Session result (from whichever provider succeeded) or BLOCKED status.

**Side Effects:** Sends Telegram alerts. May switch API provider mid-session. Logs fallback events.

## Technical Approach

`FallbackRouter` wraps the execution path:

```typescript
async function executeWithFallback(
  roster: ModelConfig,
  executePrimary: () => Promise<AgentOutput>,
  executeFallback: () => Promise<AgentOutput>,
  deps: FallbackDeps
): Promise<AgentOutput> {
  // Try primary with retries
  // On exhaustion, notify and try fallback
  // On fallback exhaustion, return BLOCKED
}
```

The `CodexAdapter` uses `@openai/codex-sdk` to run Codex sessions with native tools (file, shell, git). Codex manages its own tool execution — no MCP passthrough needed. This provides cross-provider fallback when the primary Claude path fails.

Error detection: check for HTTP 429, quota-exceeded errors, connection timeouts from the Agent SDK's error messages or thrown exceptions.
