# Task 8.6 -- Event Trigger System

## Objective

Implement the event-based trigger system that fires agents in response to git commits, task completions, and gate approvals.

## Context

In addition to cron-based scheduling, agents can be triggered by events: a commit on a specific branch, a task completing with a matching pattern, or a specific gate type being approved. The event system watches for these events and triggers the corresponding agents.

Depends on 8.1 (types) and 8.5 (shares the `triggerAgent` callback pattern from CronScheduler). Hooks into existing SessionRunner and gate system.

## Requirements

- [ ] `EventTriggerSystem` class with `EventTriggerDeps` DI interface
- [ ] `EventTriggerDeps`: `{ db: Database; triggerAgent: (def: AgentDefinition, triggeredBy: string, detail: string) => Promise<void>; repoDir: string }`
- [ ] `start(): void` -- begins watching for events
- [ ] `stop(): void` -- cleans up watchers
- [ ] `emit(eventType: string, detail: string): void` -- external call point for internal events
- [ ] Three event trigger types:
  - `on_commit: <branch>` -- triggers when a new commit lands on the specified branch
  - `on_task_complete: <pattern>` -- triggers when a task matching the glob pattern completes
  - `on_gate_approved: <gate_type>` -- triggers when a specific gate type is approved
- [ ] Commit detection: `fs.watch` on `.git/refs/heads/<branch>` file
- [ ] Task complete and gate approved: triggered via `emit()` from SessionRunner and gate resolver
- [ ] Pattern matching for `on_task_complete`: simple glob (e.g., `6.*` matches `6.1`, `6.2`, etc.)
- [ ] Deduplication: if an agent is already running (status "running" in agent_runs), skip re-trigger and log
- [ ] Registry: maintains a map of event-type definitions, updated when definitions change
- [ ] `updateDefinitions(defs: AgentDefinition[]): void` -- rebuild the event registry

## Interface Contract

**Inputs:** File system events (git refs), internal events (task complete, gate approved).

**Outputs:** `EventTriggerSystem` class.

**Side Effects:** Watches git ref files. Calls triggerAgent callback on event matches.

## What NOT to Do

- Do not modify SessionRunner or gate resolver directly -- they will call `emit()` when wired in 8.8
- Do not implement complex glob patterns -- simple `*` suffix matching is sufficient
- Do not poll git status -- use fs.watch on the specific ref file
- Do not add webhook support -- that is out of scope
