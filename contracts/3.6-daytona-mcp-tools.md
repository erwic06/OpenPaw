# Task 3.6 -- Daytona MCP Tools for Coder

## Objective

Define MCP tools that give the Coder agent file read/write, shell execution, and git operations -- all proxied through the Daytona sandbox. Uses the Agent SDK's `tool()` and `createSdkMcpServer()` to create an in-process MCP server.

## Context

The Coder agent executes in a headless Claude Code session. It needs tools to interact with the codebase, but all operations must happen inside the Daytona sandbox (design doc Section 2). This task bridges the Agent SDK tool system with the Daytona sandbox from Task 3.4. The MCP server produced here is passed to `LLMAdapter.trigger()` from Task 3.5.

Depends on Task 3.4 (Daytona sandbox manager) and Task 3.5 (Agent SDK + LLMAdapter).

## Requirements

- [ ] File tools: `file_read`, `file_write`, `file_list`
- [ ] Shell tool: `shell_exec`
- [ ] Git tools: `git_status`, `git_add`, `git_commit`, `git_push`, `git_diff`, `git_create_branch`, `git_checkout`
- [ ] All tools use `tool()` from Agent SDK with Zod schemas for inputs
- [ ] All file operations are path-scoped: reject any path outside `/workspace/<project-dir>`
- [ ] `createDaytonaToolServer(sandbox)` factory returns a configured MCP server instance
- [ ] Tools proxy operations through the Daytona SDK: `sandbox.filesystem.*`, `sandbox.process.*`, `sandbox.git.*`
- [ ] Unit tests with mocked sandbox

## Interface Contract

**Inputs:** `SandboxHandle` from Task 3.4.

**Outputs:** MCP server config passable to `query()` `mcpServers` option.

**Side Effects:** All operations execute remotely in the Daytona sandbox.

## Technical Approach

Each tool is defined with `tool()` from the Agent SDK:

```typescript
const fileRead = tool(
  "file_read",
  "Read a file from the project",
  { path: z.string().describe("Absolute path within /workspace") },
  async ({ path }) => {
    validatePath(path);
    const content = await sandbox.filesystem.downloadFile(path);
    return { content: [{ type: "text", text: content.toString("utf-8") }] };
  }
);
```

Bundle all tools into an MCP server via `createSdkMcpServer()`. The factory function accepts a `SandboxHandle` and returns the server config.

Path validation: all file paths must start with `/workspace/`. This is the enforcement mechanism from the design doc's filesystem access matrix. The `validatePath` helper rejects directory traversal attempts.

## Acceptance Criteria

- [ ] `bun run typecheck` passes
- [ ] `bun test` passes -- all new and existing tests pass
- [ ] `file_read` reads file content from sandbox
- [ ] `file_write` writes file content to sandbox
- [ ] `file_list` lists directory contents
- [ ] `shell_exec` executes commands in sandbox and returns stdout/stderr
- [ ] All git tools proxy through sandbox git operations
- [ ] Path scoping: file operations reject paths outside `/workspace/`
- [ ] `createDaytonaToolServer` returns a valid MCP server config
- [ ] No new packages added (uses `zod` and Agent SDK from Task 3.5)

## What NOT to Do

- Do not implement tools for research operations (web search, browser) -- those are Phase 4
- Do not implement tools that access the host filesystem -- sandbox only
- Do not add tools for Cubic or Vercel -- those are separate integrations
- Do not hardcode project paths -- the sandbox working directory is configurable

## Recovery Guidance

- If Daytona SDK methods have changed: check current SDK docs for filesystem/process/git method signatures
- If path validation is too restrictive: review the filesystem access matrix and adjust allowed prefixes
- If MCP server creation fails: check Agent SDK docs for `createSdkMcpServer` API changes
