# Task 5.2 -- Budget Controls

## Objective

Implement daily budget enforcement with an 80% warning alert and a 100% hard stop that pauses all headless dispatch, matching design doc failure mode F4 (lines 794-813).

## Context

The cost tracking infrastructure already exists: `logUsage` records per-session costs, `getDailySpend(db, date)` aggregates daily spend. What's missing is enforcement -- checking spend before each dispatch and pausing when the limit is hit.

The design doc specifies:
- At 80% of daily limit: alert human, continue dispatch.
- At 100%: pause all headless sessions. Running sessions finish but no new dispatch.
- Human must acknowledge (mandatory) to resume at 100%. Can raise limit, wait for reset, or switch to interactive.

Depends on Task 5.1 (AlertSystem for `budget_warning` and `budget_hard_stop` alerts). The existing `spend` gate type in `src/gates/types.ts` is reused for the mandatory acknowledgment at 100%.

## Requirements

- [ ] `BudgetConfig` type: `{ dailyLimitUsd: number; warningThresholdPct: number }` with default `{ dailyLimitUsd: 50, warningThresholdPct: 0.8 }`
- [ ] `BudgetStatus` type: `"ok" | "warning" | "exceeded"`
- [ ] `BudgetEnforcer` class with `BudgetEnforcerDeps` interface:
  - `db: Database` -- for getDailySpend queries
  - `alertSystem: { send(payload: AlertPayload): Promise<void> }`
  - `requestApprovalFn?: (request: GateRequest) => Promise<GateResult>` -- for resume gate
  - `config: BudgetConfig`
  - `nowFn?: () => Date` -- DI for testing
- [ ] `BudgetEnforcer.checkBudget(): BudgetStatus` -- queries getDailySpend, compares to thresholds
- [ ] `BudgetEnforcer.enforceBudget(): Promise<boolean>` -- returns true if dispatch should proceed:
  - If `ok`: return true
  - If `warning` and not already warned today: send `budget_warning` alert, return true
  - If `exceeded`: send `budget_hard_stop` alert, request spend gate approval, return result of gate
- [ ] SessionRunner integration: call `enforceBudget()` in `drainQueue()` before `this.queue.shift()`; if false, stop draining
- [ ] ResearchRunner integration: call `enforceBudget()` in `runResearch()` before cost estimation step
- [ ] `budgetEnforcer` added as optional field to `RunnerDeps` and `ResearchRunnerDeps`
- [ ] Warn-once behavior: only send the 80% alert once per calendar day (track last warning date)

## Interface Contract

**Inputs:** `BudgetConfig` (daily limit and warning threshold), `Database` for spend queries.

**Outputs:** `BudgetEnforcer` class exported from `src/budget/index.ts`.

**Side Effects:** Sends budget alerts via AlertSystem. Requests spend gate approval when budget exceeded. May pause headless dispatch.

## Technical Approach

```typescript
// src/budget/index.ts
export class BudgetEnforcer {
  private lastWarningDate: string | null = null;

  constructor(private deps: BudgetEnforcerDeps) {}

  checkBudget(): BudgetStatus {
    const today = (this.deps.nowFn ?? (() => new Date()))().toISOString().slice(0, 10);
    const spent = getDailySpend(this.deps.db, today);
    const { dailyLimitUsd, warningThresholdPct } = this.deps.config;

    if (spent >= dailyLimitUsd) return "exceeded";
    if (spent >= dailyLimitUsd * warningThresholdPct) return "warning";
    return "ok";
  }

  async enforceBudget(): Promise<boolean> {
    const status = this.checkBudget();
    if (status === "ok") return true;

    const today = /* ... */;
    const spent = getDailySpend(this.deps.db, today);

    if (status === "warning") {
      if (this.lastWarningDate !== today) {
        await this.deps.alertSystem.send({
          type: "budget_warning",
          dailySpendUsd: spent,
          dailyLimitUsd: this.deps.config.dailyLimitUsd,
          thresholdPct: this.deps.config.warningThresholdPct,
        });
        this.lastWarningDate = today;
      }
      return true;
    }

    // exceeded -- hard stop
    await this.deps.alertSystem.send({ type: "budget_hard_stop", dailySpendUsd: spent, dailyLimitUsd: this.deps.config.dailyLimitUsd });
    const requestFn = this.deps.requestApprovalFn ?? requestApproval;
    const result = await requestFn({
      gateType: "spend",
      taskId: null,
      sessionId: null,
      contextSummary: `Daily budget exceeded: $${spent.toFixed(2)} / $${this.deps.config.dailyLimitUsd}. Reply approve to resume dispatch.`,
    });
    return result.decision === "approved";
  }
}
```

      if (!proceed) {
        console.log("[runner] budget exceeded — dispatch paused");
        break;
      }
    }

}

- [ ] checkBudget returns "ok" when spend < 80%
- [ ] checkBudget returns "warning" when 80% <= spend < 100%
- [ ] checkBudget returns "exceeded" when spend >= 100%
- [ ] enforceBudget sends budget_warning alert at 80% (once per day)
- [ ] enforceBudget sends budget_hard_stop alert and requests spend gate at 100%
- [ ] enforceBudget returns false (pauses dispatch) when gate is denied

- [ ] SessionRunner.drainQueue stops draining when enforceBudget returns false
- [ ] ResearchRunner.runResearch returns early when enforceBudget returns false
- [ ] No new packages added

## What NOT to Do

- Do not enforce budget on interactive Claude Code sessions (those are human-managed)
- Do not kill in-progress sessions -- only prevent new dispatch
- Do not add a daily budget reset timer -- the date-based getDailySpend query handles day boundaries naturally
- Do not add configurable budget to Docker secrets in this task -- that is task 5.6

## Recovery Guidance

- If the budget enforcer breaks dispatch, the budgetEnforcer field is optional -- removing it restores previous behavior.
- If warn-once tracking causes issues, fall back to alerting on every check (noisy but correct).
