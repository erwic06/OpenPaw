# Task 2.4 -- SQLite Database Setup

## Objective

Set up the SQLite database with the schema from design doc Section 1 and provide a typed data access layer.

## Context

NanoClaw uses SQLite for operational metadata that doesn't belong in the git repo: session records, HITL gate decisions, and cost tracking. The database file lives in a mounted volume that persists across container restarts. Depends on 2.1 (container must be running).

## Requirements

- [ ] SQLite database file created at a configurable path (default: data volume)
- [ ] Schema includes all three tables from design doc Section 1: `sessions`, `hitl_gates`, `cost_log`
- [ ] Schema includes `pending_communications` table from design doc Section 3 (Gate 6)
- [ ] Schema migration runs on startup (create tables if they don't exist)
- [ ] Typed insert/query functions for each table
- [ ] WAL mode enabled for crash recovery

## Interface Contract

**Inputs:** Database file path from config/environment.

**Outputs:**
- `initDatabase(path): Database` -- initialize and migrate
- `insertSession(session): void`
- `updateSession(id, updates): void`
- `insertGate(gate): void`
- `updateGate(id, decision): void`
- `insertCostEntry(entry): void`
- `getSessionsByStatus(status): Session[]`
- `getPendingGates(): Gate[]`
- `getDailySpend(date): number`
- `insertPendingCommunication(comm): void`
- `updatePendingCommunication(id, decision, editedContent?): void`
- `getPendingCommunications(): PendingCommunication[]`

**Side Effects:** Creates/opens SQLite database file. Creates tables on first run.

## Technical Approach

Use `better-sqlite3` (synchronous, fast, well-supported) or Bun's built-in `bun:sqlite`. Schema applied via a migration function that runs `CREATE TABLE IF NOT EXISTS`. Enable WAL mode with `PRAGMA journal_mode=WAL`.

## Files to Create/Modify

- `src/db/schema.sql` -- SQL schema (all four tables from design doc Sections 1 and 3)
- `src/db/index.ts` -- Database initialization, migration, typed query functions
- `src/db/types.ts` -- TypeScript types matching the schema
- `package.json` -- Add better-sqlite3 if not using Bun's built-in

## Acceptance Criteria

- [ ] Database file is created on first run
- [ ] All four tables exist after initialization: `sessions`, `hitl_gates`, `cost_log`, `pending_communications`
- [ ] Insert and query operations work for each table (including `pending_communications`)
- [ ] WAL mode is enabled (`PRAGMA journal_mode` returns `wal`)
- [ ] Re-running initialization on an existing database is idempotent (no errors, no data loss)
- [ ] Schema matches design doc Sections 1 and 3 exactly (column names, types, constraints)

## What NOT to Do

- Do not use an ORM -- raw SQL with typed wrappers is sufficient at this scale
- Do not implement complex query patterns beyond basic CRUD -- add those when needed
- Do not store the database in the git repo (it's gitignored)
- Do not add tables beyond the Phase 2 scope (the four tables from Sections 1 and 3). Tables from Section 9 (`agent_definitions`, `agent_runs`) belong to Phase 8

## Recovery Guidance

- If better-sqlite3 has native module issues in Docker: use Bun's built-in sqlite module instead
- If WAL mode causes issues with mounted volumes: test with regular journal mode, but prefer WAL
- If schema migration fails on existing data: the migration uses IF NOT EXISTS, so it should be safe; check for column type mismatches
