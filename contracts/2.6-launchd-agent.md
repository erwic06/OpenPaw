# Task 2.6 -- launchd LaunchAgent for Auto-Start

## Objective

Create a launchd LaunchAgent plist that starts the NanoClaw Docker container on Mac Mini boot and restarts it on crash.

## Context

NanoClaw must run continuously as a daemon. On macOS, `launchd` is the standard way to manage persistent services. The LaunchAgent starts `docker compose up` on boot and restarts if the process exits unexpectedly. Depends on 2.1 (Docker container must exist).

## Requirements

- [ ] launchd plist file that starts NanoClaw via `docker compose up`
- [ ] Runs on user login (LaunchAgent, not LaunchDaemon)
- [ ] Auto-restarts on crash (KeepAlive = true)
- [ ] Logs stdout/stderr to a log file
- [ ] Does not start duplicate containers if already running

## Interface Contract

**Inputs:** docker-compose.yml location, log file path.

**Outputs:** Running NanoClaw container managed by launchd.

**Side Effects:** Registers a persistent service with macOS launchd.

## Technical Approach

Create a plist at `~/Library/LaunchAgents/com.openpaw.nanoclaw.plist`. Program: `/usr/local/bin/docker` (or wherever Docker CLI lives on the Mac Mini). Arguments: `compose -f /path/to/docker-compose.yml up`. Set KeepAlive to true. Set StandardOutPath and StandardErrorPath to log files under `~/.nanoclaw/logs/`.

Include a helper script that checks if the container is already running before starting, to avoid conflicts on manual restarts.

## Files to Create/Modify

- `com.openpaw.nanoclaw.plist` -- launchd plist file (install to ~/Library/LaunchAgents/)
- `scripts/install-launchd.sh` -- Installation script that copies plist and loads it
- `scripts/uninstall-launchd.sh` -- Uninstallation script

## Acceptance Criteria

- [ ] `launchctl load` accepts the plist without errors
- [ ] NanoClaw container starts after loading the agent
- [ ] After killing the container, launchd restarts it within 10 seconds
- [ ] Logs appear at the configured log path
- [ ] `launchctl unload` cleanly stops the service
- [ ] Running the install script on a fresh system sets up the agent correctly

## What NOT to Do

- Do not use LaunchDaemon (requires root) -- use LaunchAgent (runs as user)
- Do not implement process monitoring beyond launchd's built-in KeepAlive
- Do not create a custom init system or process manager
- Do not hardcode paths -- use variables in the install script

## Recovery Guidance

- If launchctl rejects the plist: validate XML syntax; common issue is missing required keys or wrong value types
- If container doesn't start: check that Docker Desktop is running (launchd can start before Docker Desktop on boot)
- If duplicate containers: add a pre-start check in the compose config or wrapper script
