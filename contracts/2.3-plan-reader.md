# Task 2.3 -- Plan Reader

## Objective

Implement a module that reads and parses `implementation_plan.md`, detects ready tasks, and watches for changes.

## Context

The plan reader is NanoClaw's primary input. It reads the implementation plan, parses task metadata, identifies tasks that are ready for dispatch, and triggers notifications when new tasks become available. This is the bridge between the human-edited plan and NanoClaw's automation. Depends on 2.1 (container must be running).

## Requirements

- [ ] Parse `implementation_plan.md` into structured task objects
- [ ] Extract task fields: ID, title, status, type, contract path, dependencies, assigned, artifacts, acceptance summary
- [ ] Identify tasks with `Status: ready` whose dependencies are all `complete`
- [ ] Watch the plan file for changes (file watcher or 5-minute poll interval)
- [ ] Emit events when new ready tasks are detected
- [ ] Handle malformed plan entries gracefully (log warning, skip entry)

## Interface Contract

**Inputs:** Path to `implementation_plan.md` (configured, defaults to repo root).

**Outputs:**
- `parsePlan(filePath): Task[]` -- parse plan into task array
- `getReadyTasks(tasks): Task[]` -- filter for dispatchable tasks
- `watchPlan(filePath, onReady)` -- watch for changes and call back with newly-ready tasks

**Side Effects:** File system watcher on implementation_plan.md.

## Technical Approach

Parse with regex matching the plan format:
- Task IDs: `### X.Y --` pattern
- Status: `- **Status:** (\w[\w-]*)`
- Dependencies: `- **Dependencies:** (.*)`

Use `fs.watch` or `chokidar` for file watching. Poll as fallback if fs.watch is unreliable on the mounted volume. Emit events via a simple callback or EventEmitter pattern.

## Files to Create/Modify

- `src/plan/parser.ts` -- Plan file parser (markdown to structured data)
- `src/plan/reader.ts` -- File watcher, ready-task detection, change events
- `src/plan/types.ts` -- Task type definitions
- `package.json` -- Add chokidar if needed

## Acceptance Criteria

- [ ] `parsePlan` correctly parses the current `implementation_plan.md` and returns all 6 tasks
- [ ] Each parsed task has all expected fields populated
- [ ] `getReadyTasks` returns only tasks whose status is `ready` and whose dependencies are all `complete`
- [ ] `getReadyTasks` correctly handles `Dependencies: none`
- [ ] File watcher detects when `implementation_plan.md` is modified
- [ ] Malformed entries produce a warning log but don't crash the parser
- [ ] Parser handles the `---` separator between tasks

## What NOT to Do

- Do not modify `implementation_plan.md` from this module -- it's read-only here
- Do not implement task dispatch or session management -- just reading and parsing
- Do not add NLP or fuzzy parsing -- stick to the documented regex patterns
- Do not parse contract files -- just extract the contract path

## Recovery Guidance

- If parsing fails: check that the plan file matches the expected format; common issue is inconsistent whitespace or missing field markers
- If file watcher doesn't fire: fall back to polling; Docker volume mounts can interfere with inotify
- If dependency resolution is wrong: trace through the dependency graph manually; circular dependencies should be detected and logged
