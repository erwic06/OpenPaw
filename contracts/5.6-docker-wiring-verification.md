# Task 5.6 -- Docker Wiring, Decision Logging, and Failure Mode Verification

## Objective

Wire all Phase 5 modules (AlertSystem, BudgetEnforcer, Laminar tracing) into the NanoClaw startup sequence, add decision logging to the HITL gate system, configure new Docker secrets, and verify failure modes end-to-end.

## Context

The final wiring task for Phase 5. Individual modules exist from tasks 5.1-5.5 but are not yet connected to the running system. This task:
1. Adds `laminar_api_key` and `alerts_chat_id` Docker secrets
2. Constructs and wires AlertSystem, BudgetEnforcer, and tracing in `src/index.ts`
3. Adds decision logging (JSONL append) to the HITL gate resolution path
4. Verifies Docker build and failure mode behavior

Depends on tasks 5.1-5.5. The design doc decision logging spec is at Section 5 (lines 945-966).

## Technical Approach

In `src/index.ts`, after existing Telegram setup:
```typescript
import { AlertSystem } from "./alerts/index.ts";
import { BudgetEnforcer } from "./budget/index.ts";
import { initTracing, shutdownTracing } from "./tracing/index.ts";

// Construct AlertSystem
const alertsChatId = secrets.get("alerts_chat_id");
const alertSystem = new AlertSystem({
  sendMessage,
  alertsChatId: alertsChatId ?? undefined,
  fallbackChatId: telegramChatId!,
});

// Initialize tracing
const laminarApiKey = secrets.get("laminar_api_key");
const secretValues = new Set(secrets.values());
initTracing({ laminarApiKey, secretValues });

// Construct BudgetEnforcer
const budgetEnforcer = new BudgetEnforcer({
  db,
  alertSystem,
  config: { dailyLimitUsd: 50, warningThresholdPct: 0.8 },
});

// Pass to runners
const runner = new SessionRunner({
  // ... existing deps ...
  budgetEnforcer,
  alertSystem,
});

const researchRunner = new ResearchRunner({
  // ... existing deps ...
  budgetEnforcer,
});
```

For decision logging in `src/gates/index.ts`:
```typescript
import { appendFileSync } from "fs";
import { createHash } from "crypto";

function logDecision(
  gateId: string,
  gateType: string,
  taskId: string | null,
  sessionId: string | null,
  decision: string,
  decidedAt: string,
  contextSummary: string | null,
): void {
  try {
    const entry = {
      id: gateId,
      gate_type: gateType,
      task_id: taskId,
      session_id: sessionId,
      decision,
      decided_at: decidedAt,
      context_summary_hash: contextSummary
        ? createHash("sha256").update(contextSummary).digest("hex")
        : null,
    };
    appendFileSync("/data/decisions.jsonl", JSON.stringify(entry) + "\n");
  } catch (err) {
    console.error("[gates] decision logging failed:", err);
  }
}
```

Call `logDecision()` inside `resolveGate()` after `updateGate()`.

## What NOT to Do

- Do not modify the secrets loader (`src/secrets.ts`) -- it already loads all files from /run/secrets/
- Do not hardcode budget limits in a config file -- use constants in index.ts for now (config file is a Phase 7+ concern)
- Do not add environment variables for budget config -- keep it simple with code constants
- Do not create the /data/ directory in this task -- it already exists via Docker volume mount

## Recovery Guidance

- If wiring breaks existing tests, check that all new deps are optional (budgetEnforcer?, alertSystem?). Existing test fixtures should not need these.
- If decision logging causes permission errors, check that /data/ is writable (it's mounted as `rw` in docker-compose.yml via the `nanoclaw-data` volume).
- If Docker build fails, check that @lmnr-ai/lmnr from task 5.4 is properly in bun.lock.
