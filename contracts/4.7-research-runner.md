# Task 4.7 -- Research Session Runner

## Objective

Implement `ResearchRunner` that orchestrates the full research pipeline: cost estimation, Researcher session (Gemini), Research Reviewer (Claude), and Gate 3 (research brief HITL approval).

## Context

This is the integration task for Phase 4 -- it connects all research components. The flow is: estimate cost -> confirm -> spawn Researcher -> read brief -> spawn Reviewer -> if approved -> Gate 3 -> human approves -> complete. Separate from the code `SessionRunner` because the research flow has no sandbox, git diff, code review, or deploy gate.

Depends on Tasks 4.1-4.6.

## Requirements

- [ ] `ResearchRunner` class with `ResearchRunnerDeps` interface (DI pattern)
- [ ] `runResearch(task, depth, prompt)` orchestrates the full lifecycle:
  1. Cost estimation via Haiku-tier pre-flight call
  2. Send cost estimate to Telegram, wait for confirmation
  3. Spawn Researcher session via `GeminiAdapter` with BrowserUse tool
  4. On completion, read generated brief from `research/` directory
  5. Spawn Research Reviewer via `LLMAdapter` (Sonnet)
  6. If reviewer approves: fire Gate 3 (`requestApproval` with type `"research"`)
  7. If reviewer rejects (REQUEST_CHANGES): mark task failed, include findings
  8. On human Gate 3 approval: mark task complete
  9. On Gate 3 denial/timeout: mark task blocked
- [ ] `ResearchRunnerDeps` includes:
  - `db`, `sendAlert`, `geminiApiKey`, `anthropicApiKey`
  - `browserUseDeps` (for tool wrapper)
  - `planPath` (for task status updates)
  - DI overrides: `executeResearchFn?`, `runReviewFn?`, `requestApprovalFn?`, `estimateCostFn?`
- [ ] Cost estimation module: `estimateResearchCost(deps, depth, prompt)` returns `{ estimatedTokens, estimatedCostRange }`
- [ ] Fallback routing: Gemini primary -> Claude Sonnet fallback (research tier)
- [ ] Session monitoring integration (same `SessionMonitor` pattern)
- [ ] Wire into `src/index.ts` alongside code `SessionRunner`
- [ ] Unit tests with DI mocks

## Interface Contract

**Inputs:** Task metadata, depth level (1-10), research prompt.

**Outputs:** Completed research brief (in `research/` dir), task status update.

**Side Effects:** Gemini API calls, Claude API calls, SQLite records, Telegram messages, plan file updates, Gate 3 approval request.

## Technical Approach

```typescript
export interface ResearchRunnerDeps {
  db: Database;
  sendAlert: (message: string) => Promise<void>;
  geminiApiKey: string;
  anthropicApiKey: string;
  browserUseDeps: BrowserUseDeps;
  planPath: string;
  systemPromptPath: string;       // agents/researcher/system_prompt.md
  reviewerPromptPath: string;      // agents/researcher-reviewer/system_prompt.md
  sandboxDeps?: never;             // Research has no sandbox
  // DI overrides for testing
  executeResearchFn?: (...) => Promise<AgentOutput>;
  runReviewFn?: (...) => Promise<ReviewResult | null>;
  requestApprovalFn?: (request: GateRequest) => Promise<GateResult>;
  estimateCostFn?: (...) => Promise<CostEstimate>;
}
```

Cost estimation uses a Haiku-tier call: sends the prompt + depth to a lightweight model that estimates token consumption. The estimate is formatted and sent via Telegram for confirmation before the expensive Researcher session starts.

Gate 3 (`"research"` type) is already defined in `src/gates/types.ts`. It has no timeout (brief waits for approval).
