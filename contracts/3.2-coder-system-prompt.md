# Task 3.2 -- Coder Agent System Prompt

## Objective

Create the system prompt for the Coder agent that instructs it how to operate as a headless coding session within the OpenPaw execution lifecycle.

## Context

The Coder agent is spawned by NanoClaw via the Agent SDK to execute well-specified coding tasks. Unlike interactive Claude Code sessions where a human drives the work, the Coder operates headlessly: NanoClaw pre-selects the task and provides the contract. The system prompt must adapt the 6-phase execution lifecycle from `CLAUDE.md` for this headless context.

No dependencies. File-only task (no code changes).

## Requirements

- [ ] System prompt lives at `agents/coder/system_prompt.md`
- [ ] Incorporates the 6-phase execution lifecycle (Orient, Select, Execute, Verify, Update, Terminate) adapted for headless operation:
  - Orient: read the contract provided in the prompt context
  - Select: task is pre-selected (skip -- NanoClaw provides the task)
  - Execute: implement per the contract using provided tools
  - Verify: run acceptance criteria from the contract, run tests
  - Update: update implementation_plan.md with task status, commit changes
  - Terminate: report results
- [ ] Tool usage instructions: file read/write, shell execution, git operations (all via MCP tools)
- [ ] Hard stop conditions (same as CLAUDE.md): spec needs modification, contract is wrong, task impossible, broken dependency
- [ ] Completion gates: all acceptance criteria met, tests pass, plan updated, changes committed
- [ ] Output format: structured report of what was done, what the outcome was, any issues
- [ ] Constraint: never modify `project_spec.md`, never access secrets, one task per session
- [ ] Scope boundaries: only modify files listed in the contract's "Files to Create/Modify" section unless the contract explicitly allows broader changes

## Interface Contract

**Inputs:** None (markdown file).

**Outputs:** `agents/coder/system_prompt.md` -- consumed by `LLMAdapter` when spawning Coder sessions.

**Side Effects:** None.

## Technical Approach

Write the system prompt as a markdown document. Structure it with clear sections: Role, Lifecycle, Tools, Constraints, Output Format. Reference the contract template format so the agent knows what to expect. The prompt should be specific enough to guide correct behavior but not so rigid that it prevents reasonable judgment calls during execution.

## Files to Create/Modify

- `agents/coder/system_prompt.md` -- The Coder agent system prompt

## Acceptance Criteria

- [ ] File exists at `agents/coder/system_prompt.md`
- [ ] Contains all 6 lifecycle phases adapted for headless operation
- [ ] Contains hard stop conditions matching CLAUDE.md
- [ ] Contains completion gates
- [ ] Contains tool usage instructions
- [ ] Contains output format specification
- [ ] Contains explicit constraint against modifying `project_spec.md`
- [ ] Human review: prompt is clear, unambiguous, and complete

## What NOT to Do

- Do not create system prompts for other agent types (Planner, Researcher, Reviewer) -- those are future phases
- Do not include model-specific instructions (the prompt should work with any model in the roster)
- Do not hardcode file paths that may change -- use relative references

## Recovery Guidance

- If the prompt is too vague: add concrete examples of expected behavior for common scenarios
- If agents misbehave: check the prompt for ambiguous instructions and tighten the language
