# Task 6.6 -- Project Intake Flow

## Objective

Implement the `/project` intake flow that creates a GitHub repo, scaffolds project files, runs a Planner session to flesh out the implementation plan, and presents it for mandatory plan gate approval.

## Context

The design doc (lines 1141-1149, 1651-1654) specifies: `/project` auto-creates a GitHub repo, scaffolds it (CLAUDE.md, project_spec.md, implementation_plan.md), runs a Planner session to flesh out the plan, then requires mandatory plan gate approval before any tasks are dispatched.

This task also creates the GitHub API client module (`src/github/index.ts`) using built-in `fetch` -- no octokit, per supply chain security constraints.

Depends on 6.1 (parser types), 6.2 (planner system prompt), 6.4 (dispatch). Blocks 6.9.

    - `createRepoFn?: (name, description) => Promise<{ repoUrl: string; cloneUrl: string }>` -- override for testing
    - `spawnPlannerFn?: (cwd: string, prompt: string) => Promise<string>` -- override for testing
  - `handleProject(deps, prompt): Promise<void>`
  - `deriveRepoName(prompt: string): string` helper -- extracts a kebab-case repo name from the prompt (first few words, max 40 chars)
- [ ] `/project` flow:
  1. Derive repo name from prompt
  2. Send "Creating repo..." status to Telegram
  3. Create GitHub repo via API
  4. Clone repo to `{workspacesDir}/{repoName}/`
  5. Scaffold files: `CLAUDE.md` (from template), `project_spec.md` (from prompt), `implementation_plan.md` (empty template)
  6. Git add, commit, push scaffold
  7. Send "Scaffolding complete, running Planner..." to Telegram
  8. Spawn Planner session (Claude Sonnet via Agent SDK) with `[PROJECT]` prefix and prompt
  9. Planner writes to implementation_plan.md and contracts/ in the workspace
  10. Commit and push Planner output
  11. Send plan summary to Telegram
  12. Request plan gate (`gateType: "plan"`, mandatory)
  13. On approval: send confirmation, tasks are ready for dispatch
  14. On denial: send denial message, repo remains with current state
- [ ] All exports from `src/intake/index.ts` barrel

## Interface Contract

**Inputs:** Project description prompt from `/project` command.

**Outputs:** GitHub repo created, scaffolded, Planner output committed, plan gate triggered.

**Side Effects:** Creates GitHub repo (API call), clones to local workspace, commits/pushes files, sends Telegram messages, triggers HITL gate.

## Technical Approach

```typescript
// src/github/index.ts
export interface GitHubDeps {
  githubToken: string;
  fetchFn?: typeof fetch;
}

export async function createRepo(
  deps: GitHubDeps,
  name: string,
  description: string,
  isPrivate = true,
): Promise<{ repoUrl: string; cloneUrl: string }> {
  const fetchFn = deps.fetchFn ?? fetch;
  const res = await fetchFn("https://api.github.com/user/repos", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${deps.githubToken}`,
      Accept: "application/vnd.github+json",
      "Content-Type": "application/json",
    },
    body: JSON.stringify({ name, description, private: isPrivate, auto_init: false }),
  });

  if (res.status !== 201) {
    const body = await res.text();
    throw new Error(`GitHub API error ${res.status}: ${body}`);
  }

  const data = await res.json();
  return { repoUrl: data.html_url, cloneUrl: data.clone_url };
}
```

Git operations (clone, add, commit, push) use `Bun.spawn` for shell commands, following the pattern in `src/sandbox/index.ts`. The scaffold templates are simple string literals -- CLAUDE.md references the project lifecycle, project_spec.md contains the user's prompt as the overview, implementation_plan.md is the empty template header.

Planner session: same Agent SDK pattern as `src/research/runner.ts:332-437` (Claude Sonnet, `cwd` set to workspace dir).

## What NOT to Do

- Do not use octokit or any GitHub SDK package
- Do not implement task dispatch after plan approval -- that is the normal plan watcher flow
- Do not create a general-purpose git library -- inline git operations via Bun.spawn
- Do not make the repo public by default -- private repos only
- Do not implement repo deletion or cleanup on failure -- leave partial state for manual cleanup

## Recovery Guidance

- If GitHub API auth fails, verify the token has `repo` scope
- If git push fails, check that the clone URL uses the token for auth (embed in URL or use credential helper)
- The `createRepoFn` and `spawnPlannerFn` DI overrides make testing straightforward
