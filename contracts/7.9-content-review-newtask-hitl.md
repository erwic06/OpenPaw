# Task 7.9 -- Content Review, New Task Flow, and HITL Gate UI

## Objective

Build the Content Review tab (pending communications queue with approve/edit/reject), the New Task creation dialog, and the reusable HITL gate inline UI component.

## Context

The Content Review tab surfaces Gate 6 (External Communication) items — any outbound message an agent wants to send must be approved by the human. The New Task dialog is triggered from the header's command input bar. The HITL gate inline component is reused in session panels (7.8) and the activity feed (7.7).

Depends on 7.6 (scaffold) and 7.2 (REST endpoints for communications and gates).

## Requirements

### Content Review Tab
- [ ] Content Review page at `web/src/app/content-review/page.tsx`
- [ ] Pending communications queue: each item shows content type, agent name, full content, recipient/platform, timestamp
- [ ] Three action buttons per item: [Approve] [Edit & Approve] [Reject]
- [ ] Edit & Approve opens inline editor pre-filled with draft content
- [ ] Decisions call `POST /api/communications/:id/decide`
- [ ] Recently decided section below pending queue

### New Task Dialog
- [ ] `new-task-dialog.tsx`: modal dialog triggered from header command input
- [ ] Step 1: Pick type (Research / Deep Research / Project / Coding)
- [ ] Step 2: Enter prompt (text area), select depth 1-10 for research (slider), enter project name for coding
- [ ] Step 3: Show cost estimate (calls `GET /api/cost/estimate` or uses client-side heuristic)
- [ ] Step 4: Approve → calls `POST /api/tasks` or `POST /api/projects`
- [ ] On success: navigate to relevant tab (Research or Projects)
- [ ] Validation: required fields, depth range

### HITL Gate Inline UI
- [ ] `gate-inline.tsx`: reusable component for gate approval
- [ ] Shows gate type, context summary, timestamp
- [ ] [Approve] [Edit] [Deny] buttons
- [ ] Edit mode: text area for feedback/edits
- [ ] Calls `POST /api/gates/:id/decide`
- [ ] Shows resolved state after decision (approved/denied badge)
- [ ] Used in: session panels (7.8), activity feed (7.7)

## Interface Contract

**Inputs:** REST API responses (communications, gates). User input (decisions, edits, task prompts).

**Outputs:** React components for Content Review, New Task dialog, Gate inline UI.

**Side Effects:** POST requests for communication decisions, gate decisions, and task creation.

## Technical Approach

The content editor uses a `<textarea>` with the original content pre-filled. On submit, it sends `decision: "approved_edited"` with `edited_content`. The New Task dialog uses a multi-step form with local React state managing the current step.

For cost estimation: if the backend endpoint isn't available yet (Phase 6 not wired), use a client-side heuristic based on depth level (e.g., depth 5 → ~$0.50-1.00).
