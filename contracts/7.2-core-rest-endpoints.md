# Task 7.2 -- Core Data REST Endpoints

## Objective

Implement the read-only and gate-decision REST API endpoints that the web dashboard uses to display sessions, gates, costs, projects, and pending communications.

## Context

With the router from task 7.1 in place, this task adds the bulk of the API surface. Most endpoints are read-only queries against the existing SQLite database. The gate decide endpoint hooks into the existing `resolveGate` from `src/gates/index.ts`. A `projects` table is added for the webapp to list and query projects.

Depends on 7.1. Blocks 7.3 (action endpoints), 7.7-7.9 (frontend pages).

## Requirements

- [ ] Add `projects` table to `src/db/schema.sql`: `(id TEXT PRIMARY KEY, name TEXT NOT NULL, description TEXT, repo_url TEXT, workspace_path TEXT, created_at TIMESTAMP NOT NULL)`
- [ ] Add `ProjectRow` interface to `src/db/types.ts`
- [ ] Add DB functions to `src/db/index.ts`: `insertProject`, `getProject`, `getAllProjects`, `getSessionById`, `getActiveSessions`, `getRecentSessions(limit)`, `getSessionsForTask(taskId)`, `getPendingCommunications`, `getCommunication(id)`, `updateCommunicationDecision`

## Interface Contract

**Inputs:** HTTP requests routed by the 7.1 router. SQLite database.

**Outputs:** JSON API responses. Route definitions array for registration.

**Side Effects:** `POST /api/gates/:id/decide` resolves a gate (writes to DB, triggers downstream). `POST /api/communications/:id/decide` updates communication decision in DB.

## Technical Approach

Each route file exports a function that takes `ApiDeps` and returns `Route[]`:

```typescript
// src/api/routes/sessions.ts
export function sessionRoutes(deps: ApiDeps): Route[] {
  return [
    {
      method: "GET",
      pattern: "/api/sessions",
      handler: async (ctx) => {
        const sessions = getRecentSessions(ctx.deps.db, 50);
        return jsonResponse({ data: sessions });
      },
    },
    // ...
  ];
}
```

The gate decide endpoint needs a web-compatible gate resolution path. Currently `resolveGate` in `src/gates/index.ts` is coupled to Telegram message flow. Extract the core resolution logic into a `resolveGateById(db, gateId, decision, feedback?)` function that the web endpoint and the existing Telegram flow can both call.
