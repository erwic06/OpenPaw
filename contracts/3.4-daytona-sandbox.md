# Task 3.4 -- Daytona Sandbox Manager

## Objective

Implement a module that manages Daytona sandbox lifecycle: create a sandbox per headless session, clone the project repo, provide a handle for tool operations, and destroy after session completion.

## Context

All headless code execution happens on Daytona, never on the Mac Mini (design doc Section 2). Each Coder session gets its own isolated Daytona sandbox. This module wraps the `@daytonaio/sdk` and provides the sandbox primitives that Task 3.6 wraps as MCP tools. Depends on Task 3.1 (AgentAdapter types).

## Requirements

- [ ] `createSandbox(sessionId, repoUrl, branch)` -- create a Daytona sandbox, clone repo, return a `SandboxHandle`
- [ ] `getSandbox(sessionId)` -- retrieve an active sandbox handle
- [ ] `destroySandbox(sessionId)` -- destroy the sandbox and clean up
- [ ] `SandboxHandle` exposes the Daytona SDK sandbox instance for use by MCP tools
- [ ] In-memory Map tracks active sandboxes keyed by session ID
- [ ] API key loaded from Docker secrets via `src/secrets.ts`
- [ ] DI pattern: `SandboxDeps = { apiKey: string, apiUrl?: string }` for testability
- [ ] Unit tests with mocked Daytona SDK

## Interface Contract

**Inputs:** Session ID, repo URL, branch name. Daytona API key from secrets.

**Outputs:**
- `createSandbox(...)` -- returns `SandboxHandle` with reference to Daytona sandbox
- `getSandbox(sessionId)` -- returns `SandboxHandle | undefined`
- `destroySandbox(sessionId)` -- cleanup, returns void

**Side Effects:** Creates/destroys remote Daytona sandbox environments.

## Technical Approach

Use `@daytonaio/sdk` to instantiate a `Daytona` client. On `createSandbox`: call `daytona.create()` with appropriate config, then `sandbox.git.clone(repoUrl, '/workspace', branch)`. Store the sandbox handle in an in-memory Map. On `destroySandbox`: call `daytona.delete(sandbox)` and remove from Map.

The `SandboxHandle` type wraps the Daytona sandbox and exposes the three operation categories that MCP tools need: `sandbox.filesystem.*`, `sandbox.process.*`, `sandbox.git.*`.

## What NOT to Do

- Do not implement MCP tools here -- just the sandbox lifecycle (tools are Task 3.6)
- Do not hardcode repo URLs or API endpoints
- Do not store sandbox state in SQLite -- in-memory Map is sufficient (ephemeral by design)

## Recovery Guidance

- If Daytona SDK API has changed: check the latest SDK documentation and adapt method calls
- If sandbox creation fails: verify API key is valid, check Daytona service health
- If cleanup fails: sandbox environments may need manual cleanup via Daytona dashboard
