# Task 3.10 -- Code Review (Claude Reviewer)

## Objective

After a Coder session completes successfully, spawn a focused Claude Reviewer session that reviews the git diff and returns structured findings. Replaces the planned Cubic (GitHub App) integration.

## Context

The lean integration overhaul dropped Cubic in favor of Claude-based code review. The Reviewer is a second Claude Code session with a focused adversarial system prompt, running at the `standard` tier (Sonnet) with a $0.50 budget cap. It receives the git diff in the prompt and returns structured JSON.

Depends on Task 3.6 (CodexAdapter — the Coder creates code that gets reviewed).

## Requirements

- [ ] `ReviewResult` type: `{ verdict, summary, findings[] }`
- [ ] `ReviewFinding` type: `{ severity, file, line, description }`
- [ ] `ReviewVerdict`: `"APPROVE"` | `"REQUEST_CHANGES"`
- [ ] `runCodeReview(deps, sandbox, task, diff)` — spawn Reviewer session, return structured results
- [ ] `parseReviewResult(text)` — extract JSON from session output (handles bare JSON and code-fenced JSON)
- [ ] Reviewer system prompt at `agents/reviewer/system_prompt.md` — adversarial review, structured JSON output
- [ ] Diff embedded in prompt (Approach A) — token-efficient, no tool calls for small diffs
- [ ] Reviewer gets `cwd` set to workspace for reading context files if needed
- [ ] `modelTier: "standard"` (Sonnet), `budgetUsd: 0.50` cap
- [ ] Integration in `SessionRunner.runTask()` — review step after successful Coder session
- [ ] `REQUEST_CHANGES` → task marked failed with findings
- [ ] `APPROVE` → continues to deploy gate (3.11)
- [ ] Review session failure → soft pass + alert (review failure shouldn't block shipping)
- [ ] Unit tests with mocked LLM adapter

## Interface Contract

**Inputs:** Workspace handle, task metadata, git diff string.

**Outputs:**
- `ReviewResult` — verdict, summary, findings array
- Integrated into runner: affects task final status

**Side Effects:** Spawns a Claude Code session, logs cost.

## Files to Create/Modify

- `src/review/types.ts` — `ReviewResult`, `ReviewFinding`, `ReviewVerdict`
- `src/review/index.ts` — `runCodeReview()`, `parseReviewResult()`
- `agents/reviewer/system_prompt.md` — adversarial review prompt
- `src/agents/runner.ts` — add review step after Coder completion
- `tests/review.test.ts` — unit tests

## Acceptance Criteria

- [ ] `bun run typecheck` passes
- [ ] `bun test` passes — all new and existing tests pass
- [ ] `parseReviewResult` handles valid JSON, code-fenced JSON, and malformed output
- [ ] `runCodeReview` spawns a session and returns structured review
- [ ] Runner integration: failing review prevents task completion
- [ ] Runner integration: passing review allows task completion
- [ ] Review failure handling: crashed review session does not block the task
- [ ] No new packages added

## What NOT to Do

- Do not poll any external API (no GitHub, no Cubic)
- Do not let the Reviewer explore the entire codebase — it works from the diff + referenced files only
- Do not use Opus for reviews — Sonnet is sufficient and cost-effective
- Do not block task completion if the review session itself crashes
