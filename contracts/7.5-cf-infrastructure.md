# Task 7.5 -- Cloudflare Tunnel and Access Configuration

## Objective

Configure Cloudflare Tunnel to expose the NanoClaw API at `openpaw.me` and set up Cloudflare Access with GitHub OAuth + MFA for authentication.

## Context

The web dashboard (deployed on Vercel) needs a stable URL to reach the NanoClaw API running on the Mac Mini. Cloudflare Tunnel provides a secure connection without exposing ports publicly. Cloudflare Access provides authentication — no custom auth code needed in the webapp itself. The JWT validation middleware from task 7.1 validates the CF Access JWT on every request.

Depends on 7.1 (auth middleware). This is a manual infrastructure task — the contract documents exact setup steps and provides an automation script where possible.

## Requirements

- [ ] Cloudflare Tunnel created pointing to `localhost:9999` on the Mac Mini
- [ ] DNS CNAME record: `openpaw.me` → tunnel's `.cfargotunnel.com` address
- [ ] Cloudflare Access application configured for `openpaw.me`:
  - Policy: GitHub OAuth with specific email (Eric's GitHub account)
  - MFA enabled via Cloudflare's native second-factor support
- [ ] `cloudflared` runs as either:
  - (a) Docker Compose sidecar service, or
  - (b) Host-level launchd service (like the existing NanoClaw launchd agent)
- [ ] `cf_team_domain` and `cf_audience_tag` added to Docker Compose secrets
- [ ] `src/api/auth.ts` reads team domain and audience tag from secrets for JWT validation
- [ ] Setup script at `scripts/setup-cf-tunnel.sh` documents/automates what can be automated
- [ ] WebSocket connections work through the Cloudflare Tunnel (CF Tunnel supports WS natively)

## Interface Contract

**Inputs:** Cloudflare account credentials, domain registration for openpaw.me.

**Outputs:** Working tunnel from openpaw.me to NanoClaw API. CF Access auth enforced.

**Side Effects:** DNS changes, Cloudflare dashboard configuration, new Docker secrets.

## Technical Approach

**Option A (recommended): cloudflared as Docker Compose sidecar**

```yaml
# docker-compose.yml addition
cloudflared:
  image: cloudflare/cloudflared:2024.12.2
  command: tunnel run
  environment:
    TUNNEL_TOKEN: /run/secrets/cf_tunnel_token
  secrets:
    - cf_tunnel_token
  depends_on:
    - nanoclaw
  restart: unless-stopped
```

**Option B: Host-level launchd service**

```bash
# Similar to existing NanoClaw launchd setup
brew install cloudflare/cloudflare/cloudflared
cloudflared tunnel create openpaw
cloudflared tunnel route dns openpaw openpaw.me
# Create launchd plist similar to com.openpaw.nanoclaw.plist
```

**Cloudflare Access setup (manual via CF dashboard or API):**
1. Create Access Application for `openpaw.me`
2. Add policy: Include → GitHub OAuth → specific email
3. Enable "Require MFA" in policy settings
4. Note the Application Audience (AUD) tag — used for JWT validation

## What NOT to Do

- Do not hardcode Cloudflare credentials in any file
- Do not implement custom OAuth or MFA logic — Cloudflare Access handles all of this
- Do not require the tunnel to be running for local development — auth bypass from 7.1 handles dev mode
- Do not add cloudflared as a Node/Bun dependency — it's a separate binary/container

## Recovery Guidance

- If CF Tunnel setup fails, the API still works via `localhost:9999` with auth bypass
- CF Tunnel supports WebSocket natively — no special configuration needed
- If cloudflared Docker sidecar has networking issues, try `network_mode: host` or the host launchd approach
